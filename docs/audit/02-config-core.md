# Section 02 — Config Core

## 1. Files covered

| File | Lines |
|---|---|
| `config.py` | 5,344 |
| `config_export.py` | 1,623 |
| `defaults.py` | 403 |
| **Total** | **7,370** |

`config.py` is imported (directly or via `config_export.py`) by **52 non-test
modules** and **28 test modules** (106 files total match `from config import` /
`import config` / `from config_export import` across the repo) — it is almost
certainly the single most cross-imported module in the bot. Every cog, every
scheduler loop, and both HTTP API route modules (`api/routes/guilds.py`,
`api/routes/sheets.py`) depend on it either for the `GuildConfig` dataclass,
per-feature config CRUD, the SQLite connection helper, or the Google Sheets
credential bootstrap.

## 2. Summary

Config Core is functionally solid and unusually well-documented — nearly
every function carries a docstring that explains *why*, not just *what*, and
schema/migration comments consistently reference the originating issue
number. The core risk is structural, not correctness: `config.py` is a
5,344-line god file merging six distinct responsibilities (schema DDL +
migrations, dataclass model, ~140 per-feature CRUD functions, Google Sheets
credential/I/O bootstrap, Discord-facing time/label formatting, and business
validation), and `init_db()` alone is a ~1,365-line function. The most
consequential concrete bug-class finding is that several of `config.py`'s own
helper functions perform **synchronous network I/O to Google Sheets and are
called directly from `async def` command handlers with no thread/executor
offload**, which stalls the bot's event loop for every guild during that
call. `config_export.py` is comparatively well-scoped but reaches into
`config.py`'s private `_get_conn()` helper directly in ~10 places instead of
using (or requesting) a public interface, and its `apply_import()` is itself
a ~600-line function. `defaults.py` is clean and has no findings.

## 3. Findings

### Critical

- **[Blocking I/O on event loop] `config.py:4570-4599` (`lookup_discord_id_for_name`) called from `dm.py:61-86` (`send_dm`, `mention_or_name`), both `async def`.**
  `lookup_discord_id_for_name` calls `get_member_roster_sheet()` (`config.py:4602-4622`, synchronous `gspread.authorize` + `sh.worksheet()` network calls) and then `ws.get_all_values()` (another synchronous Sheets API round-trip) — all inside a plain `def`, invoked directly (no `await`, no `asyncio.to_thread`) from two `async def` functions in `dm.py`. Every DM-driven Premium feature that resolves a Discord ID by name blocks the entire bot's event loop for the duration of a live Google Sheets HTTP call (which can be seconds under API pressure or 429 backoff).
  **Recommendation:** wrap the call site in `dm.py` with `await asyncio.to_thread(lookup_discord_id_for_name, guild_id, name)`, or make `lookup_discord_id_for_name` itself `async` and offload internally. Same treatment is needed everywhere else in the repo that calls `config.get_spreadsheet`, `config.get_spreadsheet_by_id`, `config.get_member_roster_sheet`, `config.get_or_create_worksheet`, or `config.read_member_roster_values` directly from an `async def` — a `select:"from config import" +"async def"` sweep across the ~52 importing modules is worth doing as a follow-up audit, since this finding is representative of a pattern, not an isolated bug. `read_member_roster_values` (`config.py:4645-4663`) has an in-process TTL cache that reduces frequency but does not fix the blocking nature of the underlying miss-path call.

### High

- **[God file] `config.py` is 5,344 lines mixing at least six responsibilities with no internal module boundary.** Natural seams observed while reading top to bottom:
  1. **Schema DDL + migrations** — `init_db()`, `config.py:87-1452` (~1,365 lines: `CREATE TABLE` for 20 tables, plus ~15 repeated `ALTER TABLE ADD COLUMN` migration blocks, several `DROP COLUMN` migrations, and two hand-rolled data-shape migrations with inline SQL).
  2. **Core dataclass + generic CRUD** — `GuildConfig`, `get_config`/`save_config`/`update_config_field`, `config.py:37-76, 1455-1498`.
  3. **Google Sheets credential bootstrap + I/O** — `get_spreadsheet`, `get_spreadsheet_by_id`, `get_or_create_worksheet`, `get_member_roster_sheet`, `read_member_roster_values`, `lookup_discord_id_for_name`, `describe_sheet_error`, `is_user_config_sheet_error`, `config.py:1513-1691, 4570-4663`.
  4. **Discord-facing time/label formatting** — `server_date_for`, `server_time_to_local`, `format_storm_slot`, `get_storm_slot_labels`, `get_storm_team_slot_labels`, `config.py:2040-2185` (this is presentation logic, not config storage).
  5. **~20 independent per-feature config CRUD clusters** — storm, buddy, transfer, growth, survey, birthday, member roster, shiny tasks, train, each with its own `has_X_config` / `get_X_config` / `save_X_config` / `clear_X_config` mini-API (roughly `config.py:2225` through end of file).
  6. **Domain validation logic** — e.g. `save_storm_team_plan`'s cross-team conflict + capacity validation, `config.py:2892-3002`.

  This isn't just a size smell — items 3 and 4 in particular are Discord/Sheets I/O and presentation formatting living inside what every other module imports as "the config module," which is why the blocking-I/O finding above was easy to introduce and easy to miss in review. **Recommendation (documented, not to be split now per audit scope):** the credential/Sheets-I/O cluster (3) is the cleanest extraction candidate — it has almost no coupling to the SQLite schema and could move to a `sheets_client.py` that `config.py` and everything else import from. The time/formatting cluster (4) could similarly move to a `storm_time.py` alongside `storm_date_helpers.py`, which already exists as a sibling module. The 20 per-feature CRUD clusters (5) are already internally well-isolated by comment-delimited sections (`# ── X ──`) and could become per-feature submodules (`config/storm.py`, `config/buddy.py`, …) without touching call sites if re-exported from a `config/__init__.py`.

- **[God function] `init_db()`, `config.py:87-1452` (~1,365 lines).** Beyond the size, it interleaves three different kinds of work in one body with no sub-function boundaries: (a) 20 `CREATE TABLE IF NOT EXISTS` statements, (b) ~15 blocks of `for col, definition in [...]: try: ALTER TABLE ... ADD COLUMN ... except Exception: pass` (additive migrations), and (c) at least 5 one-off *data*-migrations (column rename/drop, a poll-day-of-week formula migration at `config.py:1021-1045`, a full table-recreate migration for `storm_registration_posts` at `config.py:1328-1367`, and a JSON-payload backfill walking every row of two tables at `config.py:1378-1424`). Any new feature's schema change means editing this one function, and a mistake anywhere in it fails silently (see next finding) rather than at the call site that needed the new column.
  **Recommendation:** extract each `CREATE TABLE` into its own `_create_<table>_table(conn)` helper and each "add these N columns" migration block into a single reusable `_add_columns_if_missing(conn, table, columns: dict[str, str])` helper (see next finding) — this alone would cut the function to roughly a fifth of its current line count without changing behavior, and would make future schema changes a one-line call instead of a copy-pasted 6-line try/except block.

- **[Duplication] The `try: ALTER TABLE ... ADD COLUMN ...; conn.commit(); print(...) except Exception: pass` pattern is hand-copied ~15+ times inside `init_db()`.** Representative instances: `config.py:901-987` (storm columns, looped but the loop body itself duplicates the single-column instances elsewhere), `config.py:1048-1063` (`guild_extra_surveys`), `config.py:1069-1084` (`guild_survey_config`), `config.py:1087-1120` (`guild_train_config`), `config.py:1123-1147` (`guild_buddy_config`), `config.py:1158-1195` (`guild_transfer_config`), `config.py:1198-1210` (`guild_growth_config`), `config.py:1213-1241` (`guild_birthday_config`), `config.py:1250-1263` (`guild_configs`). Each block is structurally identical (`for col, definition in [...]: try: ALTER TABLE {table} ADD COLUMN {col} {definition}; commit; print except Exception: pass`), differing only in the table name and column list.
  **Recommendation:** factor into `_add_columns_if_missing(conn, table: str, columns: dict[str, str]) -> None` (loop + try/except/print lives once); every call site becomes a single readable line. This directly shrinks `init_db()` (previous finding) and removes ~15 places that could silently diverge in behavior (e.g. one block committing per-column, another batching — worth checking during extraction since a couple of the manually-written blocks in the 3000-3600 line range use a shared `conn.commit()` while most commit per-iteration).

- **[Error visibility] Bare `except Exception: pass` on every schema-migration `ALTER`/`DROP` in `init_db()`.** ~29 occurrences (`config.py:872, 889, 898, 986, 995, 1012, 1018, 1034, 1044, 1062, 1083, 1119, 1146, 1154, 1194, 1209, 1240, 1262, 1271, 1283, 1303, 1318, ...`). This is a deliberate, documented idiom (SQLite has no `ADD COLUMN IF NOT EXISTS`, so "already exists" is expected and silently swallowed) — but the same bare `except` also swallows a genuinely malformed `ALTER TABLE` statement (e.g. a future typo in a column type), a locked/corrupted DB file, or a disk-full error, with zero distinguishing signal. A typo'd migration would look identical to "column already exists" in the logs (or rather, would look like *nothing*, since the print only fires on the success path).
  **Recommendation:** narrow the except to `sqlite3.OperationalError` (the actual exception SQLite raises for "duplicate column name" / "no such column") and log unexpected exception types at `ERROR` instead of silently passing, so a bad migration surfaces on the next boot instead of quietly leaving a guild's schema half-migrated forever.

- **[Cross-module coupling] `config_export.py` imports `config._get_conn` (a leading-underscore, module-private helper) directly in ~10 places** instead of going through `config.py`'s public per-feature getters: `config_export.py:326, 406, 451, 497, 549, 662, 701, 977 (in the `from config import (...)` block), 992, 1137, 1216, 1271, 1316, 1383, 1493, 1530`. One instance is self-aware about this (`config_export.py:326`: `from config import _get_conn  # noqa: PLC0415 — module-private helper`), which shows the author already flagged the smell but didn't resolve it. This means `config_export.py` writes raw SQL against `guild_storm_config`, `guild_train_config`, `guild_birthday_config`, `guild_growth_config`, `guild_survey_config`, `guild_extra_surveys`, `guild_shiny_tasks_config`, and `guild_member_roster_config` schemas directly — any future column rename in `config.py` has to be grepped for and fixed in `config_export.py` too, since there's no single interface boundary that would catch it (a type checker won't catch a raw SQL column-name typo).
  **Recommendation:** add narrow public read helpers to `config.py` for the cases `config_export.py` needs "just the current value of one column for keep-current resolution" (e.g. `get_storm_log_and_post_channel_ids(guild_id, event_type)`), or accept the coupling as intentional (both files ship together, always reviewed together) but drop the `_` prefix / add a short comment in `config.py` marking `_get_conn` as an internal-but-shared helper so the noqa isn't hiding a real disagreement about the boundary.

### Medium

- **[Long function] `config_export.py:966-1564` (`apply_import`), ~598 lines.** It's a dispatcher over 9 categories (core, events, ds/cs, train, birthday, growth, surveys, shiny_tasks, member_roster), each implemented as a locally-defined closure (`_do_core`, `_do_events`, `_do_storm`, …) called through a shared `_apply(category, body)` wrapper. The closure-per-category structure is a reasonable way to get isolated try/except per category (documented intent at `config_export.py:972-975`), but at ~600 lines in one function body it's hard to unit-test a single category's apply logic in isolation and hard to scan.
  **Recommendation:** promote each `_do_X` closure to a module-level `_apply_<category>(guild_id, data, remap) -> None` function taking `data`/`remap` as explicit parameters instead of closing over them; `apply_import` then becomes a ~40-line loop over `(category, _apply_fn)` pairs. This also makes each category independently testable without exercising the whole import flow.

- **[Error visibility] `apply_import`'s per-category `_apply()` wrapper (`config_export.py:1004-1009`) catches bare `Exception` and only surfaces `str(e)` to the Discord-facing summary — nothing is logged server-side.** A real bug in, say, `_do_growth`'s JSON parsing would show up to the *user* as "growth: <exception message>" in the import summary, but wouldn't appear in bot logs / Sentry, making it hard to notice a regression across many alliances' imports unless a user reports it.
  **Recommendation:** add a `print(...)` or logger call inside `_apply()`'s except branch alongside the existing summary append, mirroring the `[CONFIG]`-prefixed logging convention already used throughout `config.py`'s migrations.

- **[Inconsistent SQL-injection guard] `config.py:1490-1498` (`update_config_field`) builds `f"UPDATE guild_configs SET {field} = ? WHERE guild_id = ?"` with no allowlist check on `field`,** unlike its three siblings elsewhere in the same file — `update_train_config_field` (`config.py:4919-4924`, checks `field not in _TRAIN_ROTATION_FIELDS: raise ValueError`), `update_buddy_config_field` (`config.py:4981-4986`, same pattern against `_BUDDY_FIELDS`), and `update_transfer_config_field`/`update_transfer_config_fields` (`config.py:5078-5116`, same pattern against `_TRANSFER_FIELDS`). Every current caller of `update_config_field` (checked via repo-wide grep) passes a hardcoded string literal, so this is not exploitable today, but it's the one config-field setter in the file that doesn't follow the file's own established defense-in-depth convention, and it's the oldest/most-called one (`guild_configs` is the core table).
  **Recommendation:** add a small allowlist derived from `GuildConfig`'s dataclass fields (`{f.name for f in dataclasses.fields(GuildConfig)}`) and validate `field` against it before interpolating, matching the pattern already proven out three times elsewhere in this same file.

- **[Duplication] Google Sheets credential-bootstrap boilerplate is copy-pasted 3 times verbatim inside `config.py`:** `get_spreadsheet` (`config.py:1513-1534`), `get_spreadsheet_by_id` (`config.py:1537-1559`), and `get_member_roster_sheet` (`config.py:4602-4622`). All three repeat the identical 10-line block (`scopes = [...]`; read `GOOGLE_CREDENTIALS_JSON` or fall back to `GOOGLE_SERVICE_ACCOUNT_FILE`; `gspread.authorize(creds)`).
  **Recommendation:** factor into a private `_gspread_client()` helper that all three call; this also gives the blocking-I/O fix (Critical finding above) a single choke point to wrap in `asyncio.to_thread` if that route is chosen instead of fixing each call site.

- **[Redundant local imports] `json` is imported at module level (`config.py:13`) but re-imported locally inside dozens of functions that don't need the late-binding-for-test-patching trick** (that pattern, per `ruff.toml:29-33`, exists specifically for `from config import X` *by other modules* re-importing a `config.*` name inside a function body so tests can monkeypatch `config.X`; it has no purpose for a stdlib module re-importing itself). Examples: `config.py:2197, 3654, 3697, 3822, 3945, 3971, 4012, 4055, 4087, 4675, 4719, 5306` (`import json` or `from datetime import ... as _dt` locally, though the datetime variants at least alias to avoid module-level `datetime`/`date` collisions, which is a legitimate reason `json` doesn't share). This is purely cosmetic noise, not a bug, but it does make it harder to tell "this local import exists for the test-patch reason" from "this local import is just habit" when scanning the file.
  **Recommendation:** low priority; if `init_db()` / the duplication cleanup above is ever tackled, sweep the redundant local `import json` statements out at the same time since `json` is never patched by tests the way `config.get_spreadsheet` etc. are.

### Low

- **[Missing return type hints] A handful of Sheets-I/O helpers lack a return type annotation** where most of the rest of the file is well-typed: `get_spreadsheet` (`config.py:1513`, returns a `gspread.Spreadsheet`), `get_spreadsheet_by_id` (`config.py:1537`), `get_member_roster_sheet` (`config.py:4602`), `get_or_create_worksheet` (`config.py:1562`, has param types but no return type). Minor since `gspread` types aren't hugely informative to annotate against, but worth a `-> "gspread.Worksheet"` / `-> "gspread.Spreadsheet"` string-annotation pass given everything else in the module is well-hinted.
- **[Docstring gap] `config_export.py`'s per-category `collect_*` functions (e.g. `collect_train`, `collect_birthday`, `collect_growth`, `collect_shiny_tasks`, `collect_member_roster`, `config_export.py:403-735`) have no docstrings**, unlike `collect_core`, `collect_events`, and `_collect_storm` earlier in the same file which do. Low priority since the function bodies are short and self-explanatory by the pattern established a few functions earlier, but it's an inconsistency within one file.
- **[Naming] `config.py`'s per-feature "field allowlist" constants use two different naming conventions**: `_TRAIN_ROTATION_FIELDS` / `_BUDDY_FIELDS` / `_TRANSFER_FIELDS` (set) alongside `_BUDDY_DEFAULTS` / `_TRANSFER_DEFAULTS` (dict, with `_FIELDS = set(_DEFAULTS)` derived from it) — buddy and transfer derive their allowlist from the defaults dict, but train's `_TRAIN_ROTATION_FIELDS` is a hand-maintained separate set at `config.py:4902-4916` with no corresponding `_TRAIN_ROTATION_DEFAULTS`. Purely stylistic; worth aligning the next time train rotation config is touched so all three follow the "derive the allowlist from the defaults dict" pattern.

## 4. What's already optimal

- **Docstring discipline is excellent and consistently answers "why," not just "what."** Nearly every one of the ~149 top-level functions in `config.py` has a docstring, and the schema comments inside `init_db()` consistently cite the originating issue number (`#251`, `#265`, `#289`, …) and explain *why* a column/migration exists, not just what it does — this is unusually good for a file this size and makes the migration history legible without needing `git blame`.
- **The "allowlist the field name before f-string-interpolating it into SQL" pattern** (`_TRAIN_ROTATION_FIELDS`, `_BUDDY_FIELDS`, `_TRANSFER_FIELDS`, `config.py:4919-4924, 4981-4986, 5078-5116`) is a genuinely good defensive pattern for the "update one dynamic column" helpers that need it — the one gap (`update_config_field`, flagged above) is the exception, not the rule.
- **`config_export.py`'s design is intentionally narrow and says so up front** (module docstring, `config_export.py:5-9`): it exports *what the alliance configured*, never member data, and explicitly has no Discord imports so the Discord UX lives in `export_import_cog.py` instead. `parse_and_validate` (`config_export.py:821-921`) is a good example of lenient-on-unknown-fields / strict-on-known-field-shape validation with user-facing, actionable error messages (line/column numbers on JSON errors, explicit "re-export from the source server" guidance on schema mismatches).
- **No hardcoded secrets, tokens, or credentials anywhere in the three files** — `GOOGLE_CREDENTIALS_JSON` / `GOOGLE_SERVICE_ACCOUNT_FILE` are both read from environment variables with a documented Railway-style fallback, and `DB_PATH` is similarly `os.getenv`-driven with a sane default.
- **Race-condition awareness is explicit where it matters.** `set_premium_assignment` (`config.py:1746-1783`) and `record_power_refresh_dm_sent` (`config.py:3407-3437`) both document *why* they rely on a UNIQUE constraint / `INSERT OR IGNORE` + rowcount rather than a check-then-write, specifically to close a race window between two concurrent Discord interactions — this is the right instinct for a system where slash-command handlers can run concurrently per guild.
- **`defaults.py` is a clean, single-purpose file with no findings** — it's exactly the "single source of truth for shippable defaults" its own docstring claims to be, consistently organized by feature with clear section banners, and the module docstring explicitly documents the house rule that defaults must stay alliance-neutral (no "us"/"our" wording), which is a nice guardrail against a specific class of regression.

## 5. Open questions

- **Is the `config.py` → `config/` package split (God file finding, High) something Kevin wants scheduled soon, or is the current "one file, clearly comment-delimited sections" shape considered acceptable given how central and stable this module is?** The per-feature CRUD clusters are already well-isolated by convention; the actual risk is mostly in `init_db()`'s size and the Sheets-I/O/formatting code living in the same file as schema/CRUD, not that the per-feature clusters are hard to find.
- **Is the blocking-Sheets-I/O-in-async-context pattern (Critical finding) already an accepted, repo-wide tradeoff** (e.g. because Discord interaction timeouts are generous enough in practice, or because guild-level command concurrency is low enough that the event-loop stall is rarely user-visible), or does Kevin want a dedicated follow-up pass across the ~52 importing modules to find every `async def` that calls a `config.py` Sheets-I/O function directly? This audit found one concrete instance (`dm.py`) by tracing a single cross-module import; a full sweep would need to check every async cog against `get_spreadsheet`, `get_spreadsheet_by_id`, `get_or_create_worksheet`, `get_member_roster_sheet`, `read_member_roster_values`, and `lookup_discord_id_for_name`.
- **Does `config_export.py` reaching into `config._get_conn` reflect an intentional "these two files are one unit, always reviewed together" design decision**, or was it just the path of least resistance when `config_export.py` was built and nobody's revisited it? If intentional, it may be worth a one-line comment in `config.py` next to `_get_conn` acknowledging the shared use, so a future reader doesn't assume the underscore prefix is being violated by accident.
