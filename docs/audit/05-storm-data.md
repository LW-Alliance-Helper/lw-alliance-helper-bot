## 2. Summary

This section is in better shape than its raw line counts suggest. `storm_attendance.py`, `storm_history.py`, `storm_member_rules.py`, `storm_roster_writeback.py`, and `storm_trends.py` are consistently disciplined: `asyncio.to_thread`/`run_in_executor` around every gspread call, module-level `logger` (not `print`), soft-error-list return conventions, and heavily-commented decision history tied to ticket numbers. `storm_roster_builder.py` at 7,226 lines is a genuine god-file by line count, but the bulk of its size is defensible domain complexity (phase-aware DS/CS assignment, paired-sub logic, draft persistence, mail rendering, image rendering, Discord posting) rather than copy-paste sprawl ‚Äî it is, however, well past the point where a future contributor can hold it in their head, and it has one real duplication pattern (triplicated `_p1`/`_p2`/`_p3` state) repeated across five-plus call sites. The most consequential finding is in `storm_log.py`: `run_log_flow` (an `async def`) calls `_prefill_from_discord_poll` directly, and that function does a blocking gspread read on the event loop with no `to_thread` wrapper ‚Äî a real event-loop-stall bug, not just a style nit. A second concrete finding is a Sheets `delete_rows` call inside a loop in `storm_roster_writeback.py`, which is a quota risk for larger rosters. There is also a real data-consistency gap between `storm_attendance.py` (writes post-#245 to the unified Member Log) and `storm_history.py` (still reads the legacy per-slot `attendance_tab`), meaning the History browser likely shows "not yet recorded" for every event recorded after the #245 cutover.

## 3. Findings

### Critical

- **`storm_log.py:1457` ‚Äî blocking gspread call on the event loop.** `run_log_flow` (line 1122, `async def`) calls `_prefill_from_discord_poll(...)` (defined line 665) directly, with no `await asyncio.to_thread(...)` / `run_in_executor(...)` wrapper. `_prefill_from_discord_poll` itself does a blocking Sheets read at lines 727-729 (`config.get_member_roster_sheet(...)` then `ws.get_all_values()`) whenever any unresolved self-vote IDs remain (line 716-732). Every other Sheets/roster read in this same function (`_collect_recent_event_dates`, `load_roster_from_config`, `count_member_flags_in_window`, `append_participation_row`, `upsert_member_log_rows`) is correctly wrapped in `run_in_executor`; this one call was missed. Because it only fires on the `roster_multi_select` + `prefill_source == "discord_poll"` path when Discord-ID votes need Member-Roster resolution, it will pass local testing silently and only stall the bot's event loop for every guild in production during a live participation-log session.
  - **Recommendation:** Wrap the call: `preselected = await asyncio.get_event_loop().run_in_executor(None, _prefill_from_discord_poll, guild_id, event_type, log_date.isoformat(), names, alias_map)`, matching the pattern used two lines above it in the same function.

- **`storm_roster_writeback.py:92-105` ‚Äî Google Sheets API call inside a loop (quota risk).** `_delete_date_rows` issues one `ws.delete_rows(idx)` call per matching row (line 105), inside `for idx in reversed(to_delete):`. This runs whenever Map Manager writes back a storm plan with `overwrite=True` for a date that already has rows ‚Äî i.e. every re-sync of an already-posted roster. A single DS/CS event roster commonly has 20-60+ rows (primaries + subs across two teams and up to three phases), so a re-sync can issue 20-60+ sequential Sheets API calls in the same request, well within range of Google Sheets' per-minute-per-user write quota, especially if multiple alliances resync around the same event window.
  - **Recommendation:** Batch the delete. gspread supports `Worksheet.batch_update` with `DeleteDimensionRequest`, or ‚Äî simpler ‚Äî build the full replacement row set in memory and do one `ws.update("A1", ...)`/`ws.clear()` + rewrite, mirroring the batched pattern already used in `storm_log.upsert_member_log_rows` (`storm_log.py:903-925`) and the header-migration rewrite in `storm_roster_builder._write_rosters_tab` (`storm_roster_builder.py:6662-6665`, which explicitly uses `ws.clear()` + single `ws.update(...)` for exactly this reason).

### High

- **`storm_history.py` vs `storm_attendance.py` ‚Äî data consistency gap (likely-stale reads).** `storm_attendance.save_attendance` (`storm_attendance.py:369-432`) documents, in its own docstring, that "Per #245 this writes ONLY to the Per-Member Log tab ‚Äî the legacy per-slot `DS Attendance` / `CS Attendance` tab is no longer appended to by the bot." But `storm_history.load_event_attendance` (`storm_history.py:195-240`) still reads exclusively from `_attendance_tab_name` (the legacy per-slot tab, line 207) ‚Äî it has no post-#245 path to the unified `<DS|CS> Member Log` tab that `storm_attendance.py` and `storm_trends.py` both read via `storm_log.read_member_log_window` / `_read_member_log_for_date`. The practical effect: any event whose attendance was recorded after the #245 cutover will render as "Attendance not yet recorded" (`storm_history.py:424-433`) in the roster-history browser, even though the attendance genuinely was recorded and is visible/queryable elsewhere (Trends Viewer, re-opening the attendance picker). This is very likely an oversight ‚Äî the #245 migration note in `storm_attendance.py` doesn't mention `storm_history.py` needing a parallel update, but the code shows it wasn't done.
  - **Recommendation:** Flag for Kevin's judgment (see Open Questions) ‚Äî either (a) point `storm_history.load_event_attendance` at `storm_log._read_member_log_for_date` / `read_member_log_window` the same way `storm_attendance.load_attendance` does, or (b) confirm this is intentional (e.g. History is deliberately being deprecated in favor of Trends) and document it inline so the next reader isn't misled by the still-present legacy-tab-reading code.

- **`storm_roster_builder.py` ‚Äî god-file by size; largest concentration of responsibilities in this section.** At 7,226 lines the file mixes: Sheets power/roster reading (lines 50-711), session state modeling (717-1073), two auto-fill algorithms (1075-1622), embed rendering (1623-2067), the main `RosterBuilderView` UI class (2074-2833, ~760 lines), zone-edit/subs-manage/assign-confirm UI (2834-3452), pairing UI (3765-4206), mail-generation/text-building (4289-5085), draft-persistence/serialization (5121-5466), DM-the-roster (5554-6041), and the Approve & Post finalize + rosters_tab writer (6042-6765). Nearly every one of these ten concerns is independently substantial (several are 400-700+ lines on their own). Per the standards doc's own precedent (`buddy.py`/`buddy_cog.py`/`buddy_hub.py`/`buddy_ui.py` split), this is a strong candidate for a future split along the seams above ‚Äî likely `storm_roster_builder_data.py` (power/roster reads + session state), `storm_roster_builder_autofill.py`, `storm_roster_builder_ui.py` (the View classes), `storm_roster_builder_mail.py`, and `storm_roster_builder_finalize.py` (Approve & Post + Sheets write).
  - **Verdict on "defensible complexity vs. tangled":** mostly defensible. The domain genuinely is this complex (phase-aware DS/CS assignment, paired vs. pool subs, power-band + per-member rules, draft persistence surviving redeploys, PNG rendering, long-mail splitting), and the code within each concern is generally well-decomposed into small, single-purpose helper functions with thorough docstrings tying back to ticket numbers. It is not spaghetti. But at this size, onboarding cost and the risk of two officers' PRs touching unrelated corners of the same file are real ‚Äî this is a "flag but do not attempt the split yourself" case per the standards doc.
  - **Recommendation:** Plan a follow-up split along the module seams listed above once a natural pause in `storm_roster_builder.py` feature work arrives; not urgent given the current decomposition quality.

- **`storm_roster_builder.py:6042-6531` ‚Äî `_finalize_structured_roster` is a ~490-line function mixing five concerns.** Power refresh, PNG render + size/attach handling, long-mail split/attachment picker, channel post (with three failure-mode branches), the Sheets write, the officer-facing summary composition, the DM-the-roster offer, and the image-overflow warning are all sequential blocks in one `async def`. This is the single largest function found across the whole section and well past the "~60-80 lines" long-function guidance, even accounting for how much of that length is comments (which are excellent) and straight-line non-branching Discord API calls.
  - **Recommendation:** Extract at minimum: `_post_roster_mail(...)` (long-mail picker + channel.send branches, lines ~6187-6337), `_build_finalize_summary(...)` (lines 6349-6415), and `_offer_dm_rostered_members(...)` (lines 6445-6489) into named helpers. Each is already a self-contained block with a single entry/exit shape, so the extraction is mechanical.

- **`storm_log.py` ‚Äî inconsistent logging: `print()` instead of `logger`.** Every other file in this section (`storm_attendance.py`, `storm_history.py`, `storm_member_rules.py`, `storm_trends.py`, `storm_roster_builder.py`) defines `logger = logging.getLogger(__name__)` and uses it. `storm_log.py` has no module logger and uses `print(...)` 14 times instead (e.g. `storm_log.py:75-78`, `367`, `380`, `731`, `876`, `927`, `941-944`, `1113-1116`, `1617`, `1686`, `1740`). `print()` output bypasses log levels, log routing/aggregation, and structured-logging configuration that the rest of the codebase relies on for production error visibility ‚Äî a Sheets-write failure in `upsert_member_log_rows` (line 927/941) is functionally invisible in any log pipeline that isn't tailing stdout directly.
  - **Recommendation:** Add `logger = logging.getLogger(__name__)` to `storm_log.py` and replace the 14 `print(...)` call sites with `logger.warning(...)`/`logger.info(...)` as appropriate (most are already warning-shaped: "X failed", "X couldn't be read").

### Medium

- **`storm_roster_builder.py` ‚Äî triplicated `_p1`/`_p2`/`_p3` phase state repeated across 5+ call sites.** `RosterBuilderSession.__init__` (717-813) carries `assignments`/`assignments_p2`/`assignments_p3`, `paired_subs`/`paired_subs_p2`/`paired_subs_p3`, and `below_floor_overrides`/`_p2`/`_p3` ‚Äî nine separate attributes for what is conceptually `dict[int, ...]` keyed 1-3. The session does provide accessor methods (`assignments_for_phase`, `paired_subs_for_phase`, `below_floor_overrides_for_phase`) that abstract most call sites, which is good ‚Äî but the same three-way enumeration re-appears verbatim in `_serialize_session` (lines 5154-5162, 5167-5185), `_apply_saved_state` (5254-5293), and `_auto_fill_session`'s reset block (1252-1261). Any future change to phase count handling (e.g. supporting 4 phases) requires hunting down every one of these triplicated blocks in addition to the accessor methods.
  - **Recommendation:** Not urgent given the accessor-method mitigation already in place, but if phase count ever needs to change, migrate the three attributes to a single `dict[int, dict[str, list[str]]]` per concern and drop the `_p2`/`_p3` naming convention.

- **`storm_roster_builder.py:517` ‚Äî module import inside a per-row loop.** `_read_roster_powers` (299-711) has `from storm_officer_view import _resolve_member_name` at line 517, inside `for row in values[1:]:` (the roster-row loop, starting line 501). Python caches the module after the first import so the runtime cost is small (a dict lookup per row), but it's still avoidable overhead on every roster read, and it obscures the function's dependencies from a quick read of the top-of-function imports.
  - **Recommendation:** Hoist to the top of `_read_roster_powers` (or the module-level lazy-import block used elsewhere in the file) alongside the existing `from storm_strategy import parse_power` at line 334.

- **`storm_trends.py:100-113` ‚Äî `query_member_log` does two separate full-tab Sheets reads where one would do.** `query_member_log` calls `storm_log.read_member_log_window(...)` (line 102) to get `distinct_dates, by_member`, then immediately calls `storm_log.count_member_flags_in_window(...)` (line 108) ‚Äî which itself calls `read_member_log_window` again internally (`storm_log.py:1059-1064`) with the same `guild_id`/`event_type`/`lookback_events`/`question_key` arguments. That's two `ws.get_all_values()` round-trips per Trends query instead of one; `count_member_flags_in_window`'s counts can be derived directly from the `by_member` dict `query_member_log` already has in hand.
  - **Recommendation:** Compute counts locally from `by_member` (already available) instead of calling `count_member_flags_in_window`, saving one Sheets read per query. Given the docstring's own note ("Quota: one Sheet read per query, no caching at v1 per the ticket"), this appears to be an unintentional regression from that stated design goal.

- **`storm_roster_builder.py:5878-5932` ‚Äî `_dm_rostered_members` sends per-member DMs via `bot.fetch_user()` + `user.send()` in a sequential loop with no batching/backoff.** For a 30-60 member alliance this is 30-60+ sequential Discord API calls per Approve & Post click. discord.py's internal rate limiter will queue/retry automatically so this won't hard-fail, but it's a candidate for the same "Discord API calls in a loop" caution the standards doc calls out, and it means a full DM fan-out can take tens of seconds with no progress indicator to the officer beyond the final ephemeral.
  - **Recommendation:** Low priority given discord.py's built-in rate-limit handling, but consider `asyncio.gather` with a small concurrency cap (e.g. `asyncio.Semaphore(5)`) to bound wall-clock time, or at minimum a "Sending‚Ä¶ (12/40)" progress edit if fan-out latency becomes a support complaint.

### Low

- **`storm_log.py` ‚Äî `_get_log_sheet` catches `gspread.WorksheetNotFound` narrowly (good) but the outer `except Exception as e` at line 70 only `print`s (see the High finging above) rather than using the module logger, undercutting the otherwise-careful distinction the docstring draws between "missing tab" and "quota/network/credential failure."** Once `storm_log.py` gets a `logger`, this specific `print` (line 75-78) should be the first one converted since the comment explicitly explains why it matters ("Falling back to the legacy tab silently could write data to the wrong sheet").
- **`storm_attendance.py:270` / `storm_attendance.py:431` ‚Äî deliberate no-op variable references (`_ = storm_log`, `_ = officer_id, prior_existing`) to silence unused-import/unused-arg linters while preserving a documented-but-currently-unused parameter/import for future use.** Not a bug, but worth a lint-suppression comment (e.g. `# noqa: F841` or a brief "kept for API stability") instead of the `_ = x` idiom, purely for readability ‚Äî a future reader has to trace back to the docstring to understand why these lines exist.
- **`storm_roster_writeback.py:87` ‚Äî `except Exception as e:  # noqa: BLE001`** is the only `# noqa` comment found in this section; harmless, but shows the file was written/reviewed with a stricter linter profile (`flake8-blind-except`) than the rest of the section appears to run under. Worth confirming whether that linter rule is meant to apply repo-wide (in which case the other broad-`except Exception` sites across this section should get the same treatment) or was a one-off.

## 4. What's already optimal

- **Blocking-I/O discipline is the norm, not the exception.** Across `storm_attendance.py`, `storm_history.py`, `storm_member_rules.py`, `storm_trends.py`, and the vast majority of `storm_roster_builder.py` and `storm_log.py`, every gspread call reachable from an `async def` is correctly wrapped in `asyncio.to_thread` or `run_in_executor`. The one miss (`storm_log.py:1457`, flagged above) stands out precisely because it's the exception to an otherwise consistently-applied pattern ‚Äî this is a codebase that clearly has a house rule about this and mostly follows it.
- **Consistent soft-error-list return convention.** Nearly every Sheet-reading function across this section returns `(data, errors: list[str])` rather than raising, letting callers degrade gracefully and surface one-line warnings to officers instead of crashing a Discord interaction. This is applied uniformly enough (`storm_attendance.load_rostered_slots`, `storm_history._read_tab_values`, `storm_member_rules.list_rules`, `storm_roster_builder._read_roster_powers`) that it reads as a deliberate house convention, not an accident.
- **Genuinely good decomposition inside the hardest algorithm in the section.** `storm_roster_builder.py`'s auto-fill (`_place_starter_in_zone`, `_fill_balanced`, `_fill_priority_greedy`, `_auto_fill_session`, lines 1075-1622) implements two distinct fill strategies over phase-aware, floor-gated, rule-pinned assignment ‚Äî genuinely intricate domain logic ‚Äî but keeps each concern in its own well-named, well-documented helper function of reasonable size (`_fill_priority_greedy` uses `itertools.groupby` cleanly rather than hand-rolled grouping). This is the section's best evidence that its size is complexity, not sprawl.
- **Ticket-linked comments throughout make intent traceable.** Nearly every non-obvious decision across all seven files cites a ticket/issue number and explains the "why," not just the "what" (e.g. `storm_attendance.py`'s Rule K / Decision #5 attendance-status history, `storm_roster_builder.py`'s #237 long-mail picker rationale, `storm_history.py`'s `_attendance_join_key` case-fold rationale). This materially reduces the risk of a future edit reintroducing a bug that was already fixed once.
- **Race-condition awareness on Sheet writes.** `storm_member_rules.delete_rule_at` (384-427) explicitly documents and defends against the two-officers-editing-concurrently race by using gspread's atomic `delete_rows` instead of read-modify-write, and re-checking row count immediately before the delete. This is a level of care not every function in the codebase demonstrates, and it's called out specifically in this section.

## 5. Open questions

- **Is `storm_history.py`'s continued read from the legacy `attendance_tab` (High finding above) an intentional deprecation-in-place, or a missed follow-up from #245?** If History is meant to be superseded by the Trends Viewer for attendance data specifically, that should be stated in `storm_history.py`'s module docstring (which currently still says "Data sources: `rosters_tab` ... + `attendance_tab` ([#133])" with no #245 caveat). If it's a miss, it's a user-visible bug (officers see "not yet recorded" for real, recorded events) that should probably be prioritized above most other findings here.
- **Is the `storm_roster_builder.py` split (High finding) worth scheduling now, or should it wait for a natural feature-driven touch of the file?** Given how well-decomposed the internals already are, a mechanical split carries real regression risk (import cycles, the "cog-import-order coupling" pattern the codebase is otherwise careful to avoid, per `storm_member_rules._strategy_helpers`'s docstring) for a payoff that's mostly onboarding/readability rather than a correctness fix. Kevin's call on priority.
- **The `storm_trends.py` double-read (Medium finding) trades a small perf/quota win against a slightly larger diff touching a Premium, officer-facing query path.** Worth confirming the Trends Viewer's real-world query volume before prioritizing ‚Äî if officers rarely click "View trends," the quota risk is theoretical.
