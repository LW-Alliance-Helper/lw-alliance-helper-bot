## 2. Summary

This is two features sharing one namespace: the original "blurb wizard" schedule (`train.py` / `train_ui.py`) and the newer Train Conductor Rotation (`train_rotation.py` / `train_rotation_ui.py`), fronted by one hub (`train_hub.py`) and one cog (`train_cog.py`). The rotation half (added later, #55) is genuinely well engineered — pure-logic/IO separation, consistent `asyncio.to_thread` offloading, dataclasses, and a birthday-conflict dedup mechanism that was clearly hardened after a real production incident (#89). The legacy blurb half (`train.py` / `train_ui.py`) predates that discipline and has not been brought up to the same bar: it makes blocking Google Sheets calls directly inside `async def` Discord callbacks with no offload, and `train_ui.py` does date-range filtering business logic that belongs in `train.py`. The `_cog`/`_hub`/`_ui` split is followed in spirit but not applied evenly — the newest module (`train_rotation_ui.py`) is the model, the oldest (`train_ui.py`) is the outlier.

## 3. Findings

### Critical

- **[Blocking I/O on the event loop] `train_ui.py:198,206,301,311,370,386,499,525,561`** — Every write path in the legacy blurb UI calls `load_schedule()` / `save_schedule()` (both do synchronous `gspread` HTTP calls, see `train.py:66-154`) directly inside `async def` callbacks, with no `asyncio.to_thread` / executor offload. This includes `run_blurb_wizard_for_entry` (the multi-minute interactive wizard), `AddEntryModal.on_submit`, `UpdateEntryModal.on_submit`, and all three `TrainActionView` button handlers (`update`, `generate`, `clear`). Each call stalls the entire bot's event loop — every guild, every command — for the duration of the Sheets round-trip. Compare to `train_hub.py`, which wraps the same functions in `asyncio.to_thread` everywhere (e.g. `train_hub.py:244-245,257,295,297,300`), and to `train_rotation_ui.py`, which offloads consistently via `asyncio.to_thread`/`*_async` wrappers throughout. **Recommendation:** wrap every `load_schedule`/`save_schedule`/`mark_blurb_generated` call in `train_ui.py` with `asyncio.to_thread`, matching the pattern already used in `train_hub.py`.

- **[Blocking I/O in a per-minute background loop] `train_cog.py:503,509,514,563,658`** — `TrainCog.check_reminder` (a `@tasks.loop(minutes=1)` that iterates every guild the bot is in) calls `load_schedule(guild.id)`, `check_and_add_birthdays(...)`, `save_schedule(...)`, and `load_birthdays(tab_name, guild.id)` directly (no offload) inside the loop body. This is worse than the `train_ui.py` case because it runs automatically every minute for every configured guild, not just on a user-triggered interaction — a slow/rate-limited Sheets response here freezes the bot for every other guild's commands too. The same file already knows the correct pattern one function later: `_maybe_post_daily_confirm` (`train_cog.py:839-841`) wraps its Sheets read in `loop.run_in_executor`. **Recommendation:** wrap the four calls above in `asyncio.to_thread`/`run_in_executor`, consistent with `_maybe_post_daily_confirm` in the same file.

### High

- **[Fragility pattern recurs after being fixed once] `train_cog.py:317-319,736-739,783-793,815-829`** — The Train Conductor Rotation's daily loop dedups the weekly-draft and daily-confirmation posts with **in-memory-only** sets (`self.rotation_draft_fired`, `self.rotation_confirm_fired`), reset only on a date change. This is the exact bug class that was already found and fixed for birthday auto-population: the code comment at `train_cog.py:305-310` explains that `birthday_population_fired` used to be an in-memory set, Railway restarts wiped it, and it "re-fired and spammed conflict messages on every redeploy" — the fix was moving the dedup into the database (`guild_birthday_config.last_train_population_date`, see `get_birthday_population_last_fired`/`mark_birthday_population_fired`, `train_cog.py:490-493,526`). The rotation loop was written after that fix (#55 postdates #89) but did not adopt the same DB-backed pattern. A Railway restart at the exact draft/confirm minute will re-fire: for the daily confirmation this reposts a duplicate confirmation view; for the **weekly draft** it's worse — `_maybe_post_weekly_draft` calls `regenerate_week_async`, which discards and re-rolls every conductor for the week, silently overwriting any manual edits leadership already made that day. **Recommendation:** persist `last_rotation_draft_date` / `last_rotation_confirm_date` per guild in the DB (mirroring `last_train_population_date`) and check it before firing, the same way the birthday path now does.

- **[UI file doing business logic] `train_ui.py:499-521,523-549`** — `TrainActionView.update()` and `TrainActionView.generate()` build the entry list directly in the button-callback body: load the schedule, iterate `schedule.items()`, parse each ISO date, filter by a date-range cutoff, and sort — duplicated near-identically between the two methods (only the window differs: -7d/+30d vs 0d/+14d). This is exactly the "UI doing business logic" pattern the `_cog`/`_hub`/`_ui` split exists to avoid, and it isn't unit-testable without spinning up a `discord.ui.View`. Contrast with `train_rotation.py`, which keeps all such filtering/selection pure and importable (e.g. `rotation_counts`, `member_tally`). **Recommendation:** extract a `get_updatable_entries(schedule, *, days_back, days_forward)`-style helper into `train.py`, call it from both button handlers.

### Medium

- **[Duplicated leadership-role check] `train.py:583-587` vs `train_rotation_ui.py:210-217`** — `_is_leadership()` and `_is_leader()` are two independently-maintained copies of the same three-line "does this user have the configured leadership role" check (`get_config(guild_id)` → compare `cfg.leadership_role_name` against the interaction user's role names). A future change to how leadership is resolved (e.g. supporting multiple roles) has to be made in both places, and nothing enforces that. **Recommendation:** move one canonical version into `config.py` (or a small shared `auth.py`) and import it from both.

- **[Duplicated date-parsing systems] `train.py:219-324` vs `train_birthdays.py:40-217`** — `train.py` defines `MONTH_MAP` + `parse_date_and_name()` and `train_birthdays.py` independently defines `_BIRTHDAY_MONTH_MAP` + `parse_birthday()`. Both parse a human-typed "month name + day" string into a calendar date, with separately hand-maintained month-abbreviation dictionaries that have already drifted (`train_birthdays.py`'s map includes `"sept": 9` as a 4-letter form; `train.py`'s does not). They're not drop-in identical (one returns a name+theme-hint, the other a bare (month, day) tuple, and the birthday parser is more permissive about separators/ordinals/years), so a straight merge isn't free, but the month-name table itself is pure duplication and a candidate typo (a new alias added to one map and not the other) is easy to introduce unnoticed. **Recommendation:** at minimum, extract a single shared `MONTH_MAP` constant both modules import.

- **[Long function mixing three concerns] `train_cog.py:450-718` (`check_reminder`, ~270 lines)** — One `@tasks.loop` body handles birthday auto-population into the schedule, Discord birthday announcements + DMs, and the legacy train daily reminder + DM, each with its own per-guild loop and its own time-gate parsing, nested up to 5 levels deep (`for guild → try → if bcfg → if hour/minute → if not-already-fired → try`). Standards flag both the length (way past the ~60-80 line guideance) and the nesting. **Recommendation:** split into three private helpers (`_run_birthday_population`, `_send_birthday_announcements`, `_send_train_reminder`), called in sequence from a slim `check_reminder`, each with its own early-return guard clauses instead of nested `if`s.

- **[Long function mixing input/business/output] `train_ui.py:36-224` (`run_blurb_wizard_for_entry`, ~190 lines)** — One function owns interactive step-through (theme/tone/notes/template prompts), cancellation-event races via `asyncio.wait`, Sheets persistence, and final prompt formatting + posting. It's more forgivable here since the steps are inherently sequential/stateful, but the persistence block (`train_ui.py:197-206`) and the prompt-build-and-post block (`208-220`) are cleanly separable from the conversational steps above them. **Recommendation:** at minimum extract the final "persist + build + post" tail into its own helper so it can be tested/reused without re-running the whole wizard.

### Low

- **[Unused logger] `train_hub.py:20,29`** — `import logging` + `logger = logging.getLogger(__name__)` is set up but `logger` is never called anywhere in the file (verified via `grep -n "logger\."`); every other message in the file uses `print(...)`, matching the rest of the section (`train.py`, `train_cog.py`, `train_rotation.py` all use `print`). Either the logger is dead code, or it's the intended direction and the rest of the file (and section) hasn't been migrated to it. **Recommendation:** remove the unused `logger`, or — if `logging` is meant to replace `print` in this codebase — file that as its own cleanup rather than leaving one half-migrated call site.

- **[Duplicated date-parameter alias] `train.py:610` and `train_cog.py:66`** — Both files independently define `date_cls = date` to avoid the `date` slash-command-parameter name shadowing the `datetime.date` type, with an identical explanatory comment copy-pasted. Harmless, but it's the same one-line workaround maintained in two places. **Recommendation:** no action needed unless a third file needs it — noting only because a grep for `date_cls` will surprise a future reader into thinking there's one canonical alias.

- **[Duplicated day-persist helpers] `train_rotation_ui.py`: `WeeklyDraftView._persist_day` (`~1485-1497`) vs `DailyConfirmView._persist_scheduled` (`~1936-1948`)** — Both are a 10-line "load train config → call `tr.set_day_status(..., status=STATUS_SCHEDULED)`" wrapper, differing only in reading `dd`/`self.dd`. Minor; both call sites already correctly wrap the call in `asyncio.to_thread`. **Recommendation:** low priority — could become a `tr`-level `save_scheduled_day(guild_id, tab, dd)` helper taking a `DraftDay` directly, removing both wrappers.

## 4. What's already optimal

- **`train_rotation.py` is a model pure-logic/IO split.** The selection algorithm (`select_conductor`, `_lowest_rotation`, `generate_week_draft`, `reroll_day`), the fairness counting (`rotation_counts`, `leaderboard`, `member_tally`), and the birthday pre-pass (`_place_birthdays`) are all plain functions over dataclasses with zero Discord or even I/O dependency — genuinely unit-testable, and the module's own docstring explains this is deliberate. The Sheet I/O half is clearly demarcated below a banner comment and funnels through one shared `_rewrite()`/`_open_tab()` pair instead of each function reinventing clear-and-rewrite.

- **`train_rotation_ui.py` and `train_hub.py` consistently offload blocking Sheets calls.** Every gspread-touching call in these two files is wrapped in `asyncio.to_thread` (or a `*_async` wrapper that resolves premium status then offloads, e.g. `load_rotation_state_async`, `regenerate_week_async`), including inside button callbacks, modals, and the background-loop dispatch functions. This is the correct pattern and should be the one the rest of the section (`train_ui.py`, `train_cog.py`'s `check_reminder`) is brought up to.

- **The birthday↔train conflict logic is a genuine post-incident hardening success story.** `train_birthdays.py:250-443` shows real engineering response to a specific production complaint: `CONFLICT_HANDLED_WINDOW` widens the "already handled" check past the ±1-day placement window specifically because leadership manually resolves conflicts a few days off; `conflict_key()` gives a stable, Discord-ID-first dedup identity so an "Ignore" click silences exactly one occurrence without suppressing next year's; and `mark_conflict_ignored`/`get_ignored_conflict_keys` persist that dismissal in the DB rather than in memory. `BirthdayConflictView` in `train_cog.py` replaced a "fire-and-forget text alert" with interactive resolution buttons, and the accompanying comments make the reasoning explicit. This is exactly the kind of thing #334 flagged as missing — and it's already fixed here, which makes the High finding above (the rotation loop's in-memory-only dedup) more notable, since it's the *same* fix not yet propagated to the newer feature.

- **No bare `except:`, no mutable default arguments, and no unlogged swallowed exceptions found** across all seven files (checked via grep for `except:` and `except Exception:` followed by a silent `pass`). Every `except Exception as e:` site in this section logs via `print(f"[...] ... {e}")` with a module tag prefix, which at least keeps failures visible in logs even though the section doesn't use structured logging.

## 5. Open questions

- **Is the in-memory `rotation_draft_fired`/`rotation_confirm_fired` gap (the High finding above) already known and accepted as low-risk?** The comment at `train_cog.py:315-316` says "robust outage catch-up is #227's job" for a different concern (loop heartbeat), which suggests outage/restart handling for this loop is tracked somewhere — worth checking whether #227 (or a follow-up) already covers the redeploy-at-the-exact-minute case, or whether it's a genuine gap Kevin wants filed.
- **Is `train_ui.py` slated for retirement rather than fix-up?** The legacy blurb surface (non-rotation guilds) and the newer rotation surface are two different mental models bolted under one `/train` hub. If most guilds are expected to migrate to rotation over time, it may be more valuable to plan `train_ui.py`'s deprecation than to invest in bringing its blocking-I/O pattern up to the rotation code's bar — worth Kevin's call before spending effort there.
- **Should `train_rotation_ui.py`'s size (2017 lines) get a seam identified now?** Per the standards doc, this is a "look for natural seams for a future split, don't do it yourself" file. Candidate seams observed while mapping it: the preset editor (`TrainPresetEditorView` + its pickers, roughly lines 659-1334), the weekly draft surface (`WeeklyDraftView` + embed builders, ~1335-1807), and the daily confirmation surface (`DailyConfirmView` + `_ConfirmPostModal`, ~1808-2017) are each fairly self-contained and could become `train_rotation_ui_presets.py` / `_draft.py` / `_confirm.py` if this file keeps growing. Flagging for Kevin's judgment on whether it's worth doing now versus at the next feature addition.
