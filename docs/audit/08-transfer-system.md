# Transfer System Audit

## 1. Files covered

| File | Lines | Role |
|---|---|---|
| `transfer.py` | 1037 | Pure-logic core: column addressing, filter DSL, change detection, templates. No `discord`/`gspread` imports. |
| `transfer_cog.py` | 767 | Poll loop (`tasks.loop`), notification embeds/views, source-copy orchestration, `/transfers` command registration. |
| `transfer_setup.py` | 3442 | The "Setup Transfers" wizard: mode picker, column-mapping UI, filter builder, decisions manager, re-entry edit menu. |
| `transfer_sheets.py` | 164 | Thin gspread I/O layer (read/write/validate) shared by the wizard and the poll loop. |
| `transfers_hub.py` | 281 | `/transfers` front-door hub: status embed + View/Check-now/Setup buttons. |

## 2. Summary

This is the healthiest-looking large feature seen so far in this sweep. It already follows the `_logic` / `_cog` / `_ui` / `_sheets` split the standards doc asks for (`transfer.py` is deliberately kept `discord`/`gspread`-free and "trivially unit-testable"; `transfer_sheets.py` isolates all Google Sheets I/O). Every Sheets call across all four Discord-facing files is wrapped in `asyncio.to_thread` and a `try/except` that surfaces a friendly message via `config.describe_sheet_error` — there is no blocking-call-on-event-loop risk and no silent-failure risk anywhere in this section. `transfer_setup.py` is the one file that's a real god-file candidate purely on size (3442 lines), and it has genuine, fixable duplication in its column-picker view classes and its `run_transfer_setup` orchestrator function, but it is not a tangled mess — it's a large, consistent set of small, similarly-shaped wizard steps that could be mechanically split.

## 3. Findings

### High

- **[Duplication] `transfer_setup.py`: three near-identical paged multi-select view classes.** `_ColumnMapView` (593–767), `_AdaptiveColumnMapView` (805–1014), and `_SourceMapView` (1578–1706) each reimplement the same boilerplate: per-page `_render()`, `interaction_check`, `_on_name` (clear-if-on-page logic), `_on_prev`/`_on_next` (page clamp + `_render()` + `safe_edit_response`), and a `column_map()` builder. Concretely, `_headers_for`/`column_map` are byte-for-byte identical between `_ColumnMapView` (transfer_setup.py:738-754) and `_AdaptiveColumnMapView` (transfer_setup.py:866-882); `_SourceMapView` (transfer_setup.py:1635-1690) reimplements the same page-nav + name-select + column_map pattern a third time with a narrower field set. Recommendation: extract a `_PagedColumnPickerView` base (page state, `_render`/`_on_prev`/`_on_next`/`interaction_check`/`_headers_for`) that the three subclasses customize only with which selects they add and what `column_map()` returns.

- **[Structure] `transfer_setup.py` is 3442 lines — a strong split signal.** Per the standards doc's >2000-line threshold, this is well past the point where natural seams should be identified for a future split (not attempted here). The file already telegraphs its own seams via its section comments: entry/select-helpers (1–225), sheet-entry + mode picker (226–590), column mapping (590–1065), channel/style/filter steps (1066–1575), intake-source configuration (1575–1956), decisions/write-back (2096–2497), the linear wizard driver (2497–2870), and the re-entry edit menu + sub-menus (2872–3442) — almost exactly the shape of a `transfer_setup_sheets.py` / `transfer_setup_mapping.py` / `transfer_setup_filter.py` / `transfer_setup_decisions.py` / `transfer_setup_wizard.py` / `transfer_setup_editmenu.py` split. Recommendation: flag for a future split along those existing comment-delimited seams; the re-entry edit menu (2872–3442, ~570 lines) alone is a clean, self-contained extraction candidate since it only calls into the other sections' public step functions.

### Medium

- **[Long function] `run_transfer_setup` (transfer_setup.py:2500–2818) is ~320 lines.** It linearly drives every step of first-time setup (mode → sheets → mapping → channel → style → frequency → filter → intake sources → enrich toggle → templates → decisions → removal → finalize), each guarded by a repeated `if x == "CANCEL": return` / `if x == "TIMEOUT": ...` block. It's readable *within* itself (one step per paragraph, consistently shaped), but it is well past the "~60-80 lines" guideline and mixes orchestration with config-persistence calls (`config.update_transfer_config_field(s)`) inline at every step. Recommendation: at minimum extract the per-mode sheet/mapping dispatch (2541–2638, ~100 lines) and the tail finalize block (2795–2816) into named helpers; a fuller decomposition could give each step a `(status, value)`-returning wrapper that also persists, cutting the boilerplate at every call site.

- **[Duplication] Repeated "read sheet, catch, friendly-error, bail" block.** The exact 4-6 line pattern (`asyncio.to_thread(transfer_sheets.read_sheet, sheet_id, tab)` wrapped in `try/except Exception as e: await channel.send(f"⚠️ Couldn't read your sheet: {config.describe_sheet_error(e)}"); return "OK"`) appears at transfer_setup.py:3072-3076, 3159-3163, and in slightly different shape inside `_SheetStepView._verify` (297-311) and `_ModeStepView._verify` (501-533) — 4+ near-duplicate call sites. Recommendation: a small `_read_sheet_or_notify(channel, sheet_id, tab) -> (header, rows) | None` helper (posting its own error message and returning `None` on failure) would collapse all of these to one line each.

- **[Duplication] `_ModeStepView._verify` (transfer_setup.py:501-549) substantially duplicates `_SheetStepView._verify` (297-335).** Both: extract the sheet id from a possibly-pasted URL, read the sheet in a thread, catch and report the same two failure shapes (unreadable sheet, missing header row), then disable the view's buttons and edit the message with a "✅ … Continuing below." confirmation. `_ModeStepView._verify` is the general form (loops over 1-2 named sheet roles); `_SheetStepView._verify` is the same logic hardcoded to one sheet. Recommendation: have `_SheetStepView` call `_ModeStepView`'s multi-sheet verify logic (or extract both onto a shared free function) with a single-item role list, rather than maintaining two copies of the error text.

- **[Long function] `_build_filter` (transfer_setup.py:1475-1572) is ~100 lines** mixing the initial none/build/keep gate, a `while True` clause-building loop, join-tracking (`joins[i]` connecting `clauses[i]`/`clauses[i+1]`), and final shape normalization (`{"and": [...]}` vs `{"clauses": ..., "joins": ...}`) in one body. It's well-commented and correct, but is a decomposition candidate — e.g. split the clause-loop into its own `_collect_filter_clauses` helper returning `(clauses, joins)` or a CANCEL/TIMEOUT sentinel.

- **[Typing] Public/cross-module functions in `transfer_setup.py` mostly lack type hints on parameters.** `run_transfer_setup(interaction: discord.Interaction, bot)` (called from `transfers_hub.py:189` and `setup_cog.py`) and `_launch_transfer_setup(interaction: discord.Interaction, bot)` (called from `transfers_hub.py:189`) hint `interaction` but not `bot`'s type or `-> None`. This is a stark contrast with `transfer.py`, whose every function is fully hinted including `list`/`dict`/`str | None` generics, and with `transfer_cog.copy_sources(cfg: dict, target_header: list) -> dict`. Recommendation: at minimum hint the two cross-module entry points; the ~50 private wizard-step helpers are lower priority per the standards doc's "especially cross-module" carve-out.

### Low

- **[Duplication, repo-wide idiom] The `if view.cancelled: return "CANCEL"` / `if not view.confirmed: return "TIMEOUT"` pair appears 31 times in `transfer_setup.py`.** This is a documented, deliberate repo-wide convention (see `wizard_registry.wait_view_or_cancel`'s docstring, which prescribes exactly this call shape), so it isn't a standards violation — but at 31 call sites in one file it's a lot of boilerplate. A tiny wrapper like `async def _await_or_bail(view, cancel_event) -> str | None` returning `"CANCEL"`/`"TIMEOUT"`/`None` would let call sites do `if (r := await _await_or_bail(view, cancel_event)): return r` and cut ~2 lines per site. Flagging as Low since changing a repo-wide idiom has cross-file blast radius outside this section.

- **[Docstrings] A handful of the ~50 private step functions in `transfer_setup.py` (e.g. `_edit_channel`, `_edit_style`, `_edit_frequency`, transfer_setup.py:3102-3148) have no docstring**, unlike most of their siblings (`_edit_mapping`, `_edit_filter`, `_edit_intake` all have one). Minor and low-priority per the standards doc's private-helper carve-out, noted only because the surrounding file is otherwise docstring-dense.

- **[Naming] `transfer.py:_norm_header` and `transfer_sheets.py:_norm`** are two separately-defined private normalization helpers with different names doing conceptually the same "trim + casefold" job on header/column strings (transfer.py:86-91 vs transfer_sheets.py:82-83), plus `transfer_cog.py` reaches into `transfer._norm_header` directly at transfer_cog.py:192 (a private, underscore-prefixed cross-module import). Not a bug — `transfer_sheets._norm` also handles `None` input which `transfer._norm_header` also does, so they're near-duplicates rather than divergent — but worth consolidating on one shared name, and `transfer_cog.py` reaching into `transfer._norm_header` (a "private-looking helper", per the standards doc's cross-module-coupling category) would be cleaner as a public `transfer.norm_header` or a small `transfer.header_col_index(header, name)` helper that returns the index outright instead of making the caller reach in.

## 4. What's already optimal

- **No blocking-call risk anywhere in this section.** Every one of the ~20 gspread-touching call sites across `transfer_cog.py`, `transfer_setup.py`, and `transfers_hub.py` is wrapped in `asyncio.to_thread`, with `transfer_sheets.py` itself kept a synchronous, gspread-only module by design (per its own module docstring) precisely so the offload boundary is unambiguous.
- **No error-visibility gaps.** Every Sheets read/write is inside a `try/except Exception as e: # noqa: BLE001` that logs (`logger.warning`), reports to Sentry via the shared `_capture` helper (transfer_cog.py:59-64), and surfaces a user-facing message via the shared `config.describe_sheet_error(e)` formatter — this directly addresses the kind of Sheets `APIError 400/503` failures referenced by issues #356/#350; there's no place in this section where a Sheets failure would silently vanish or crash the poll loop. The poll loop additionally backs off to the configured interval on a read failure (transfer_cog.py:517-524) instead of hot-looping against a broken sheet.
- **Real separation of concerns.** `transfer.py`'s module docstring explicitly states the design rule ("It imports nothing from `discord` so it stays trivially unit-testable") and the code honors it — zero `discord`/`gspread` imports in a 1037-line module that does real domain logic (filter DSL, identity hashing, poll diffing). This is the cleanest instance of the `_logic`/`_cog`/`_ui` split referenced in the standards doc seen in this feature area.
- **Consistent, well-labeled "degrade soft" behavior in the filter/diff logic.** `_eval_clause`, `cell_for`, and `status_snapshot` all have explicit, documented fallbacks (unaddressable column → pass the filter rather than drop the applicant) rather than throwing or silently mis-filtering — a deliberate, well-reasoned design choice called out in both the module docstring and inline comments.
- **The re-entry edit menu (`_edit_*` family, transfer_setup.py:3063-3301) is DRY at the per-section level** even though the file overall is large: each editor follows the same `_edit_gate` (show current, Keep/Change) → do the work → confirm pattern, making it easy to verify each one is correct by inspection rather than needing to trace unique control flow per section.

## 5. Open questions

- **Is a `transfer_setup.py` split worth doing given how coherent it already is?** The file is well past the 2000-line "look for seams" threshold, and this doc identifies clean split points (§3, High), but the module is unusually well-organized internally (consistent step-function shapes, heavy comments, one clear `run_transfer_setup` driver). Kevin's call on whether the maintainability payoff of a split is worth the churn of updating every internal reference, versus leaving it as one large-but-navigable file.
- **Worth generalizing the three paged-picker view classes (§3, High) now, or wait until a fourth similar picker is needed?** The duplication is real and mechanical (three copies of the same page-nav logic), but `_ColumnMapView`/`_AdaptiveColumnMapView`/`_SourceMapView` do have genuinely different field sets and save-validation messages, so the extraction isn't purely mechanical copy-paste — it needs a judgment call on how much to parameterize a shared base vs. how much duplication is "good enough" for three call sites.
- **`_norm_header` vs `_norm` consolidation (§3, Low) touches `transfer.py`'s "no discord/gspread" boundary rule.** `transfer_sheets.py` intentionally doesn't import from `transfer.py` in the reverse direction anywhere obvious, but it easily *could* import `transfer._norm_header` (or a promoted public version) without pulling in `discord`. Worth confirming there isn't a deliberate reason `transfer_sheets.py` keeps its own copy before merging them.
