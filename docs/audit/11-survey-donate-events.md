# Section 11: Survey, Donate & Events Hub

## 1. Files covered

- `survey.py` — 1932 lines
- `donate.py` — 816 lines
- `events_hub.py` — 1146 lines

## 2. Summary

Three loosely-related standalone features, each owned by one file with no meaningful coupling between them (confirmed: `donate.py` shares nothing with the other two; `events_hub.py` correctly reuses `scheduler.next_event_dates`/`post_editor` rather than duplicating scheduling logic). Correctness and async hygiene are solid across the board — no bare excepts, no mutable defaults, no unwrapped blocking Sheets calls, no Sheets-API-in-a-loop risk. The main cost center is size: `run_survey()` in `survey.py` and `_run_create_event_wizard()` in `events_hub.py` are both single functions doing the job of an entire wizard (300+ and 290+ lines respectively), and both files show a repeated "ask-with-retry" / "disable-buttons-and-stop" shape that's copy-pasted rather than shared. `donate.py` is the cleanest of the three and is a good model for the other two (see Positives).

## 3. Findings

### Critical

None found.

### High

- **[Structure] `survey.py` mixes UI, business logic, and Sheets I/O in one 1932-line file.** The file combines persistent Discord UI (`SurveyButtonView`, `DynamicSurveyButton`, `DropdownView`, `_SurveyPickView`, `_SurveyManageView`, `_ReminderHubView`, `_DestinationPickView`, `_ChannelPickView`, `_FrequencyPickView`, `_DayPickView`), the survey question-asking business logic (`run_survey` and its nested `ask_*` closures), the numeric-magnitude parsing domain logic (`_parse_magnitude_input`, `_fmt_response_value`), Sheets I/O (`update_squad_powers`, `append_survey_history`), and a `tasks.loop` scheduler tick (`check_scheduled_reminders`) all in one module. Per the repo's own `buddy.py`/`buddy_cog.py`/`buddy_hub.py`/`buddy_ui.py` split, this is a candidate to separate into `survey_logic.py` (parsing + sheets I/O), `survey_ui.py` (views/modals), and `survey_cog.py` (commands + scheduler tick). Not urgent at 1932 lines, but it's the single largest file in this section and is >2x the "warrants a look" threshold.

- **[Function length] `run_survey()` is a ~360-line function (survey.py:269-633) that is itself a small wizard framework.** It defines five nested async closures (`ask_number` 295-329, `ask_dropdown` 331-342, `ask_numeric` 344-430, `ask_multi_select` 432-468, `ask_date` 470-510), each mixing input-prompt formatting, `bot.wait_for` I/O, validation, and re-prompt-on-failure retry logic, followed by the main question loop (514-559), the Sheets save step (561-590), and the leadership-notification embed build (592-631). Recommend extracting the five `ask_*` helpers to module level (they don't need `thread`/`user`/`check` as closures — pass them as params) and splitting the save + notify steps into their own functions (e.g. `_save_survey_response()`, `_notify_leadership()`).

- **[Function length] `_run_create_event_wizard()` is a ~290-line function (events_hub.py:740-1033).** It walks Name → Time → Schedule (repeating/manual) → Anchor Date → Interval → Blurb → Save in one function body, with two locally-defined View classes (`_ScheduleView`, `_BlurbChoiceView`) nested inside it. Same shape as `run_survey()` above — recommend splitting each wizard step into its own helper (`_ask_event_name`, `_ask_event_time`, `_ask_event_schedule`, `_ask_event_blurb`) called in sequence from a slim orchestrator, matching the step pattern already used well in `survey.py`'s `_run_schedule_wizard` (which at least delegates to `_ask_time`/`_ask_reminder_message` instead of inlining everything).

- **[Duplication] Near-identical next-event-date rendering logic duplicated within `events_hub.py`.** `_build_events_hub_embed()` (lines 197-238) and `_render_upcoming_followup()` (lines 495-537) both iterate `events`, branch on `schedule_type == "repeating"`, call `scheduler.next_event_dates()`, compute the same `days = (nxt - today).days` / "today"/"tomorrow"/"in N days" phrasing, and handle the same `(ValueError, TypeError)` anchor-parse failure — with only the final string format differing. Recommend extracting a shared `_next_event_line(ev, today) -> str` (or a `(date, label)` tuple) used by both call sites so a future change to the "next date" phrasing or bug fix doesn't need to be made twice.

### Medium

- **[Duplication] "Disable buttons, edit response, stop()" boilerplate repeated ~10+ times in `survey.py`.** The exact three-line pattern `for item in self.children: item.disabled = True` → `await wizard_registry.safe_edit_response(inter, view=self)` → `self.stop()` (or the interaction-response variant) appears verbatim in `DropdownView._cb` (254-260), `ask_multi_select`'s `_cb` (451-456), `_SurveyPickView._cb` (835-845), `_SurveyManageView` (three button handlers, 899-926), `_ReminderHubView` (three handlers, 1354-1376), `_DestinationPickView` (two handlers, 1418-1442), `_ChannelPickView._cb` (1457-1465), `_FrequencyPickView` (three handlers, 1555-1577), and `_DayPickView._cb` (1592-1598). Recommend a small base class (`_SingleChoiceView` or a `disable_and_finish(self, inter, content=None)` mixin method) that every one of these views inherits/calls instead of re-typing the three lines each time.

- **[Duplication] Retry-loop "ask, validate, decrement attempts_left, re-prompt" shape repeated 4x in `survey.py`.** `ask_number` (295-329), `ask_numeric` (344-430), `ask_date` (470-510), and (in spirit) the "too many attempts" wording, all repeat the same `attempts_left = 5` / `while attempts_left > 0` / timeout-then-cancel skeleton with only the validation step differing. A generic `_ask_with_retry(thread, check, bot, validator, max_attempts=5)` helper would collapse ~120 lines of near-identical control flow into one place, and the events_hub.py time-parsing loop (861-873, see next finding) could potentially reuse the same shape if lifted to a shared module.

- **[Duplication, cross-file] Manual `HH:MM` fallback-parse check duplicated verbatim between `survey.py` and `events_hub.py`.** `survey.py:1845-1846` and `events_hub.py:861-863` both contain the identical `if len(raw) == 5 and raw[2] == ":" and raw.replace(":", "").isdigit(): return raw` fallback after calling `setup_cog._parse_12h_time()`. Since both already import `_parse_12h_time` from `setup_cog`, this fallback belongs next to it (e.g. `setup_cog.parse_time_flexible()` that does both steps) rather than being copy-pasted at each call site.

- **[Error visibility] Silent `except Exception: cfg = None` / `events = []` in `_build_events_hub_embed()` (events_hub.py:166-173).** If `config.get_config()` or `config.get_guild_events()` raises for a reason other than "not configured yet" (e.g. a DB connectivity blip), the hub silently renders as if nothing were configured, with no log line. An officer would see "not set" everywhere and have no signal that something is actually broken vs. just unconfigured. Recommend at minimum a `logger.warning(...)` in each except block, consistent with the `logger.warning` already used at line 439-442 in the same file.

- **[Error visibility] Silent `except Exception: continue` while parsing `reminder_time` in `check_scheduled_reminders()` (survey.py:1049-1052).** If a scheduled-reminder row has a malformed `reminder_time`, the entry is skipped every single minute forever with zero logging — leadership has no way to discover why their reminder never fires. Recommend logging the guild_id/survey_id and the bad value before `continue`, similar to the pattern already used a few lines below at 1116-1121 for the outer per-entry exception handler.

- **[Rate-limit awareness] `_send_reminder_via_dm()` DMs roster members serially in a plain `for` loop (survey.py:1329-1341).** Each `await dm.send_dm_to_id(...)` is awaited one at a time with no batching, no concurrency limit, and no explicit backoff on 429s (discord.py's internal bucket limiter will throttle automatically, but a large roster will still take a long time to fully DM since nothing is parallelized). Not a correctness bug, but for a large alliance this could make `/survey remind` (DM path) or the scheduled-DM tick take noticeably long. Worth a `asyncio.gather` with a small `asyncio.Semaphore` if roster sizes grow.

- **[Consistency] `events_hub.py` inconsistently offloads its own blocking config/DB calls.** `handle_events_hub()` wraps `_build_events_hub_embed(guild)` in `asyncio.to_thread(...)` (line 1134) specifically because it calls `config.get_config`/`config.get_guild_events` (both synchronous SQLite reads via `config._get_conn()`), but every other call site in the same file that makes the identical `get_config`/`get_guild_events` calls — `_open_today_editor` (326-327), `_render_upcoming_followup` (483), `_render_log_followup` (553), `_open_create_picker` (675), `_run_create_event_wizard` (772, 839), `_open_delete_picker` (1045, 1062) — calls them directly with no `to_thread` wrap. Either the offload is unnecessary everywhere (SQLite local reads are typically sub-millisecond) or it's needed everywhere; the current split-the-difference approach reads as accidental rather than deliberate. Worth a decision one way or the other rather than one lone `to_thread` call.

- **[Cross-module consistency] `survey.py`'s question-answer wizard doesn't use `wizard_registry`'s cancel/registration flow that `events_hub.py`'s create-event wizard uses.** `_run_create_event_wizard()` registers a `cancel_event` via `wizard_registry.register(user.id)` and threads it through every `ask_text`/view-wait call so the flow can be cleanly cancelled and unregistered. `run_survey()`'s own `ask_number`/`ask_numeric`/`ask_dropdown`/etc. have no equivalent — they only handle `asyncio.TimeoutError`, with no shared-registry cancellation path. This may be intentional (survey answering is member-facing, not an officer wizard), but it means the two "walk the user through N questions" flows in this section behave differently under cancellation. Flagged here for Kevin's call — see Open Questions.

### Low

- **[Dead code] Unused imports in `survey.py`.** `import json` (line 23) and `import os` (line 24) are both unused — no `json.` or `os.` reference anywhere in the file (confirmed via grep). Safe to remove.

- **[Typing] `guild_id: int = None` pattern instead of `int | None = None`.** `update_squad_powers()` (survey.py:157) and `append_survey_history()` (survey.py:200) both type `guild_id: int = None`, which is technically incorrect (the type hint says `int` but the runtime default is `None`). Both are cross-module functions (imported/patched in `tests/unit/test_survey.py`, `tests/sheets/test_sheet_writes.py`). Same pattern in `_get_spreadsheet(guild_id: int = None)` (survey.py:150).

- **[Typing] `bot` parameter left untyped throughout both `survey.py` and `events_hub.py`.** e.g. `run_survey(bot, thread, user, survey=None)` (survey.py:269), `_send_reminder_to_channel(bot, guild_id, channel_id, body)` (survey.py:1276), `_open_today_editor(bot, interaction)` (events_hub.py:318), `_run_create_event_wizard(bot, interaction, *, preset=None)` (events_hub.py:740-745). A `bot: commands.Bot` (or the project's `discord.Client` subtype) annotation would help IDE/type-checking on these cross-module entry points; low priority since it's a repo-wide habit, not unique to this section.

- **[Naming/consistency] `survey.py` and `donate.py` use `print(...)` for operational logging while `events_hub.py` uses the stdlib `logging` module (`logger = logging.getLogger(__name__)`).** e.g. `survey.py:192`, `:233`, `:589`, `:631`, `:994-997`, `:1028`, `:1089-1121` all use `print(f"[SURVEY] ...")`; `donate.py:759-764` uses `print(f"[PREMIUM] ...")`. `events_hub.py` uses `logger.warning`/`logger.info` (lines 440, 1033, 1078-1082). Whichever is the intended house style, the three files in this section don't agree with each other — worth standardizing (`logging` is strictly better for log-level filtering and Railway log routing).

- **[Swallowed exception, low-risk] `except Exception: pass` around `ws.set_basic_filter()` (survey.py:224-227).** Cosmetic Sheets formatting call; failure here is genuinely inconsequential, but a one-line log would cost nothing and help debug "why doesn't my history tab have a filter" reports.

- **[Typing] `DropdownView.__init__(self, placeholder: str, options: list, label: str = "")` (survey.py:242) — `options: list` has no element type (`list[str]`).** Minor; same for a few other `options: list` params in `ask_dropdown`/`ask_multi_select`.

## 4. What's already optimal

- **Sheets I/O is correctly offloaded and N+1-safe.** `update_squad_powers()`/`append_survey_history()` are synchronous gspread functions, and the *only* call sites (survey.py:566-584, verified via grep) wrap them in `loop.run_in_executor(None, ...)` rather than awaiting them directly on the event loop. Within `update_squad_powers()`, the whole sheet is read once via `get_all_values()` and the matching row is found with a Python scan rather than issuing one API call per row — this avoids the N+1-Sheets-API-call trap the task asked to check for. No Sheets-API-in-a-loop risk was found anywhere in this section.

- **`events_hub.py` correctly reuses `scheduler.py` rather than reimplementing it.** All next-event-date math goes through `from scheduler import next_event_dates` (events_hub.py:155, 322, 481) and the actual draft-posting UI goes through `from scheduler import post_editor` (322, 360, 454) — there is no parallel/competing scheduling implementation in this file, which is exactly the boundary the task asked to verify.

- **`donate.py`'s `_ConfirmActionView` is a well-factored shared confirm/cancel component** (lines 91-136) — a single parameterized view (`confirm_label`, `confirm_style`) reused by `/premium assign` (fresh-assign and switch-guild paths) and `/premium unassign`, including an `interaction_check` that pins the view to the invoking user. This is exactly the kind of shared base class `survey.py`'s many one-off confirm/pick views (see Medium duplication finding) should be following.

- **Race handling in `donate.py`'s premium-assignment flow is thoughtful.** `/upgrade`, `/premium assign`, and `on_entitlement_create` all re-check `premium.get_assigned_user(guild_id)` immediately before writing, and handle the "another subscriber claimed it between check and confirm" race with a specific, user-facing error message rather than silently overwriting or crashing (e.g. donate.py:282-296, 532-546, 719-735).

- **Docstrings are consistently present on public/cross-module functions** across all three files — `run_survey`, `update_squad_powers`, `append_survey_history`, `_parse_magnitude_input`, `_build_events_hub_embed`, `handle_events_hub`, every `DonateCog` command — with real explanations of *why*, not just restating the signature.

## 5. Open questions

- Should `run_survey()`'s member-facing question flow adopt the same `wizard_registry` cancel/registration pattern that `_run_create_event_wizard()` (and other officer wizards in `setup_cog.py`) use? Right now a member answering a survey has no explicit "cancel" path other than letting it time out — worth confirming whether that's an intentional UX difference (member flow vs. officer flow) or an oversight.
- Is the `print()` vs. `logging` split across `survey.py`/`donate.py` (print) vs. `events_hub.py` (logging) a deliberate in-progress migration, or should this section standardize now? Affects how much of the "Low: naming/consistency" finding is worth acting on before a later sweep.
- The `_build_events_hub_embed` vs. every-other-call-site inconsistency around `asyncio.to_thread`-wrapping the SQLite-backed `config.get_config`/`get_guild_events` calls: is offloading these considered necessary at all in this codebase, or was the one `to_thread` call in `handle_events_hub` just defensive copy-paste from a different (actually-slow) pattern elsewhere? Resolving this one way clears up the Medium consistency finding above.
- `donate.py`'s `_pending_upgrade_guilds: dict[int, int]` is in-memory, per-cog-instance state bridging `/upgrade` and `on_entitlement_create` across an arbitrary time gap (however long Discord checkout takes). The code comments already acknowledge the Railway-restart edge case is an accepted tradeoff — flagging only to confirm that's still the desired behavior rather than, say, persisting the pending guild to the DB alongside the assignment table.
