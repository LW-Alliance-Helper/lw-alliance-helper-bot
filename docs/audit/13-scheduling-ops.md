## 2. Summary

This section is architecturally split between an older, dynamic-sleep central scheduler (`scheduler.py`, purpose-built for the leadership event-draft/announcement flow) and a newer generation of per-feature `@tasks.loop(minutes=1)` cogs (Storm, Train, Survey, Shiny — living outside this section but cross-referenced below) that all report into the well-designed `outage_catchup.py` recovery system via a shared heartbeat + adapter registry. `outage_catchup.py`, `release_announcements.py`, `stats_publisher.py`, and `support_join_watch.py` are genuinely strong: good docstrings, tz-aware datetimes throughout, per-item error isolation, and proper `logging` usage. The most serious finding is that the exact timezone bug class fixed in the live loops for shiny tasks (#330/#331) and the train reminder (#318) — using the guild's local calendar date instead of the Last War server (UTC-2) date at the reset boundary — has been reintroduced in `outage_catchup.py`'s recovery path for both surfaces, because its `_scheduled_today` helper computes "today" from guild-local time and none of the `scan_shiny`/`scan_train_reminder` adapters route through `config.server_date_for`. `scheduler.py` itself is the weakest file in the set: it still uses `print()` instead of `logging`, has a data-loss gap around its in-memory 5-minute-warning queue, a stale/misleading type hint, a blocking synchronous DB call inside its main loop, and one very long nested handler.

## 3. Findings

### Critical

- **[Timezone / correctness] `outage_catchup.py:207` (`scan_shiny`) and `outage_catchup.py:446` (`scan_train_reminder`) recompute "today" from the guild's local calendar date, not the Last War server (UTC-2) date, reintroducing the exact bug class fixed in the live loops (#330/#331 for shiny, #318 for train).** Both adapters call `scheduled = _scheduled_today(window, tz, *parsed)` (`outage_catchup.py:151-155`), where `_scheduled_today` derives `today` via `window.end.astimezone(tz).date()` — the guild's own timezone, not server time. That date is then used directly as the in-game day: `scan_shiny` passes `today=scheduled.date()` into `build_announcement_for_guild` (`outage_catchup.py:226`), and `scan_train_reminder` builds `today_str = scheduled.date().isoformat()` (`outage_catchup.py:446`) to key `schedule.get(today_str)`. Compare this to the live loops, which both explicitly route through `config.server_date_for` for exactly this reason: `bot.py:933` (`shiny_today = server_date_for(guild_now)`, with a comment citing #330) and `train_cog.py:657` (`today_str = server_date_for(guild_now).isoformat()`, comment citing #318). For any guild whose local evening clock time is past the UTC-2 server reset but before local midnight — precisely the scenario those two issues were about — an outage that straddles that window will cause the recovery digest to compute the wrong in-game day: it can offer to post yesterday's shiny-server list, or fail to find today's train conductor (or find the wrong one), even though the live loops would get it right.
  - **Recommendation**: change `scan_shiny` and `scan_train_reminder` to resolve the in-game date via `config.server_date_for(scheduled)` (or `server_date_for(window.end)`) instead of `scheduled.date()`, matching the live loops. Consider moving the "in-game date" resolution into `_scheduled_today` itself (or a sibling helper) so every current and future adapter that needs the LW server day gets it by construction rather than by each adapter remembering to opt in.

- **[Data loss] `scheduler.py`'s 5-minute-warning queue (`pending_warnings`, `scheduler.py:49`) is in-memory only and is not covered by any `outage_catchup.py` surface adapter, so a restart between "announcement approved" and "5-minute warning due" silently drops the warning with no recovery path.** `ApprovalView._post_to_announcements` (`scheduler.py:750-756`) schedules the warning purely by writing into the module-level `pending_warnings` dict; `run_scheduler` (`scheduler.py:1017-1028`) is the only reader. There is no persistence (no DB row, no file) and `outage_catchup.py`'s `SURFACE_ADAPTERS` tuple (`outage_catchup.py:631-638`) only lists `scan_event_draft, scan_shiny, scan_survey, scan_birthday, scan_train_reminder, scan_storm_signup` — no warning-queue scanner. A Railway redeploy landing in the (up to) 5-minute window between approval and warning fire time means the promised "5-minute warning auto-fires" behavior (module docstring, `scheduler.py:17`) just doesn't happen, with no digest, no log-visible recovery offer, nothing — which is a worse failure mode than the surfaces `outage_catchup.py` was specifically built to fix.
  - **Recommendation**: either persist `pending_warnings` (a small SQLite table, mirroring the pattern `config.py` already uses for other scheduled state) so it survives a restart, or add a `scan_pending_warning` adapter to `outage_catchup.py` that reconstructs pending warnings from the announcement post history. At minimum, log a startup warning enumerating any warnings that were pending at the moment of a clean shutdown so the gap is visible instead of silent.

### High

- **[Blocking I/O] `scheduler.py:989-991` opens a raw synchronous `sqlite3.connect(DB_PATH)` connection and runs a blocking query directly inside `run_scheduler`'s `async def` loop body, with no `asyncio.to_thread`/executor offload.** This is the central scheduler's own main loop (the one function everything else in this section defers to for the leadership-approval flow), so any stall here — a slow disk, WAL contention with another writer — blocks the event loop for every guild, every command, and every other cog for the duration of the query. The exact same pattern also exists in `bot.py:715` (`growth_task`), so this is systemic rather than unique to this file, but `scheduler.py` is the highest-frequency, most central instance of it in this section.
  - **Recommendation**: wrap the `sqlite3.connect(...)...fetchall()` block in `await asyncio.to_thread(...)`, or add a small `config.get_active_guild_configs()` helper that does the offload once and is reused by both `scheduler.py` and `bot.py:growth_task`. Given `config.py` appears to make this same synchronous-sqlite-from-async-code tradeoff everywhere (see Open Questions), a single shared offloaded accessor would fix both call sites in this section at once.

- **[Duplication] The Last War "server time" UTC-2 offset is defined independently in two places that must never drift apart.** `shiny_tasks.py:38` defines `_SERVER_TZ = timezone(timedelta(hours=-2))` (used by `_creation_date_from_ms`), while `config.py:2037` separately defines `SERVER_TZ_OFFSET = -2` (used by `server_date_for`). Both encode the same game-defined constant, but neither imports the other, so a future change to the offset (or a DST-handling fix, however unlikely for a fixed-offset game clock) would have to be applied in two files to stay consistent — and nothing would flag a partial update.
  - **Recommendation**: define the offset once (e.g. `config.SERVER_TZ_OFFSET` as the canonical source, or a shared `server_time.py`) and have `shiny_tasks.py` import it rather than redeclaring `_SERVER_TZ` locally.

### Medium

- **[Error visibility] `scheduler.py:90-96` `_resolve_event_info` swallows every exception from `get_guild_event` with a bare `except Exception: ev = None`, no logging at all.** Unlike most of this file's other except blocks (which at least `print(...)`), a real bug here (bad guild config row, DB error) is completely invisible — the function just silently falls back to legacy `EVENT_LIBRARY` behavior for that event.
  - **Recommendation**: log the exception (even at `debug`/`info` level, since a missing guild-configured event is an expected fallback path) so a systemic DB problem doesn't hide behind a "just fell back to legacy" symptom.

- **[Typing] `scheduler.py:49` `pending_warnings: dict[str, datetime]` is typed as mapping to a bare `datetime`, but every write actually stores a 3-tuple.** `ApprovalView._post_to_announcements` (`scheduler.py:753`) does `pending_warnings[self.event_key] = (warn_dt, self.event_list, self.guild_id)`, and the reader at `scheduler.py:1018-1019` unpacks accordingly (`warn_dt, event_list, guild_id = val`). The declared type hint is simply wrong, which defeats its purpose for anyone using it for static checking or as documentation.
  - **Recommendation**: correct the annotation to `dict[str, tuple[datetime, list[dict], int]]` (or introduce a small `PendingWarning` dataclass, which would also make the tuple-unpacking at the call sites self-documenting).

- **[Long function / nesting] `scheduler.py:388-492` `EventEditorView.add_event`'s inner `on_select` closure is ~100 lines and mixes: modal-less text-prompt collection (`wait_for`), time parsing, per-event timezone resolution, message cleanup/teardown, list mutation, and UI refresh with its own nested try/except/logging.** This is the longest single unit of control flow in the section and the hardest to safely modify — a change to any one concern (e.g. the timezone resolution) requires reading through unrelated cleanup and refresh logic to find it.
  - **Recommendation**: extract at least the "collect + parse a time from chat" flow (prompt → `wait_for` → parse → cleanup) into a shared `_prompt_for_time(interaction, channel, label) -> tuple[int,int] | None` helper — it is near-identical to the same flow already duplicated in `edit_time`'s `on_select` (`scheduler.py:528-543`) — leaving `add_event`'s closure to own only the event-specific list mutation and refresh.

- **[Duplication] The "prompt for a time in chat, `wait_for`, parse, clean up the messages" sequence is duplicated between `add_event`'s `on_select` (`scheduler.py:396-431`) and `edit_time`'s `on_select` (`scheduler.py:528-556`)**, differing only in the prompt text and what happens to the parsed result.
  - **Recommendation**: same fix as above — a shared `_prompt_for_time` helper collapses both call sites and would also fix any future time-parsing bug in exactly one place instead of two.

- **[Long function] `release_announcements.py:155-236` `maybe_post_release_announcement` is ~80 lines with seven distinct early-return branches (no metadata row, no version change, no content authored, opted out, setup incomplete, channel missing, success) plus its own `try/except discord.Forbidden`/`except Exception` wrapper.** Every branch is well-commented and individually simple, so this is a soft flag rather than a real risk — the same style choice (heavily commented early returns) reads fine in place — but it's at the upper edge of the standards' 60-80 line guidance.
  - **Recommendation**: no urgent action; if the function grows another branch (e.g. a per-guild rate limit), consider extracting the "should we post, and if not why" decision into a pure `_release_decision(cfg, meta, current_version) -> Decision` helper that the Discord-side function then acts on, mirroring how `should_announce` is already split out.

### Low

- **[Logging inconsistency] `scheduler.py` uses `print(f"[SCHEDULER] ...")` / `print(f"[SCHEDULER][ERROR] ...")` throughout (e.g. lines 687, 744, 754-756, 929, 1042-1049, 1080-1083, 1099-1109, 1121-1125) while every other file in this section (`outage_catchup.py`, `release_announcements.py`) sets up `logger = logging.getLogger(__name__)` and uses it.** `print` output bypasses log levels and any centralized log routing/filtering the bot has configured elsewhere.
  - **Recommendation**: add `logger = logging.getLogger(__name__)` to `scheduler.py` and replace the `print(...)` calls with `logger.info(...)`/`logger.warning(...)`/`logger.error(...)` as appropriate, matching the newer files in this same section.
- **[Style] `scheduler.py:479` uses `logger = __import__("logging").getLogger(__name__)` inline inside an exception handler instead of a normal top-level `import logging`.** Functionally fine, but unusual and easy to miss on a read-through; it's also the only `logging` usage anywhere in the file (see the Logging inconsistency finding above), so it reads as a one-off patch rather than an intentional choice.
  - **Recommendation**: fold into the `scheduler.py`-wide `logging` adoption above — a single top-level `import logging` / `logger = logging.getLogger(__name__)` removes the need for the `__import__` call entirely.
- **[Dead code / duplication] `scheduler.py` imports `from config import get_config` twice at module scope** — once at line 28 and again, immediately after a comment header, at line 35 — with no change in between. Harmless but redundant.
  - **Recommendation**: delete the second `from config import get_config` (line 35).
- **[Error visibility] `outage_catchup.py:792-796` and `outage_catchup.py:822-826` (`OutageCatchupView._fire_and_report`, `.dismiss`) both wrap the final digest-message edit in a bare `except Exception: pass` with no logging.** By this point the recovery items have already fired (or been dismissed), so the blast radius of a swallowed failure is small — it only affects whether the digest message itself gets an updated summary line — but it's the one silent-except pattern in an otherwise well-logged file.
  - **Recommendation**: swap `pass` for a one-line `logger.debug(...)` so a pattern of failed digest-message edits (e.g. a permissions change mid-flight) would show up if anyone goes looking.

## 4. What's already optimal

- **`outage_catchup.py`'s detection + adapter-registry design is a strong piece of infrastructure.** The heartbeat-gap detection (`detect_outage_window`) is intentionally scoped to only the four reliable per-minute loops (documented explicitly, `outage_catchup.py:18-27`, with the reasoning for excluding the variable-sleep `scheduler.py` loop spelled out both there and in `scheduler.py:973-978` — the two files agree with each other on this, which is exactly the kind of cross-module contract that's easy to let drift). Each surface is a self-contained adapter function with a uniform `(bot, guild, cfg, window) -> list[MissedItem]` signature, registered in one tuple (`SURFACE_ADAPTERS`, `outage_catchup.py:631-638`), and `_scan_guild` (`outage_catchup.py:644-658`) isolates each adapter's failure with `try/except` + `logger.exception` so one broken surface can't sink the whole recovery digest for a guild. This is a textbook extensible-registry pattern and should be the template other clock-driven surfaces (see the Critical/data-loss finding on `pending_warnings` above) are folded into.
- **Every timestamp-bearing dataclass and function signature in `outage_catchup.py` is explicitly tz-aware**, and the module docstring calls out the ordering hazard around heartbeat snapshotting (`snapshot_heartbeats()` must run before `on_ready`'s loops start) clearly enough that a future maintainer is unlikely to reintroduce the race. `OutageWindow`, `_scheduled_today`, `_was_missed`, and `fmt_clock` all take/return `datetime` objects with explicit `tzinfo`, and naive datetimes are never compared against aware ones.
- **`shiny_tasks.py`'s server-time math is well-reasoned and well-tested at the unit level**: `is_shiny_today`'s docstring cites the exact verification method used (cross-checked against cpt-hedge for specific server numbers, `shiny_tasks.py:92`), and `_creation_date_from_ms` explicitly resolves the UTC-2 server date rather than the naive UTC date, with the #331 rationale spelled out inline (`shiny_tasks.py:33-38`). The live-loop half of the #330/#331 fix (`bot.py:933`) is genuinely correct — it's specifically the `outage_catchup.py` recovery path that didn't inherit the fix (see Critical finding above).
- **`release_announcements.py` and `stats_publisher.py` both model "must never crash the caller" correctly.** `release_announcements.maybe_post_release_announcement` wraps its entire body in `try/except discord.Forbidden` (with a Sentry breadcrumb, not a full capture, since perms drift is alliance-owned) followed by `except Exception` (full `sentry_sdk.capture_exception`), so a bad guild never aborts the `on_ready` loop for the rest. `stats_publisher.publish_alliance_count` likewise swallows and logs every network/parse failure and is explicitly documented as "must NEVER take down the bot" — and it backs that up by diffing against the existing file content before committing, so a redeploy-heavy day doesn't spam the website repo's git history with no-op commits.
- **`support_join_watch.py` is a clean example of the "Discord-thin" split the standards doc asks for**: `shared_bot_guilds`, `format_join_notice`, and `verified_role_blocker` are all plain, fully type-hinted, unit-testable functions with no side effects, and the module docstring explicitly states this is deliberate so `bot.py` only owns the listener wiring. `verified_role_blocker` in particular is a good example of ordering a validation chain by user-actionability (permission gap reported before hierarchy gap) with the reasoning stated inline.

## 5. Open questions

- The synchronous-`sqlite3`-from-async-code pattern flagged above (`scheduler.py:989-991`) appears to be how essentially all of `config.py` talks to SQLite — no `asyncio.to_thread` usage was found anywhere in `config.py` itself. Is that an accepted, deliberate tradeoff (local SQLite reads are sub-millisecond in practice, so the event-loop stall is negligible even under load), or is it worth a repo-wide pass to offload the hottest call sites? This section can only fix its own call sites; the systemic question is bigger than Scheduling & Background Ops.
- `scheduler.py`'s central loop and the newer per-feature `@tasks.loop(minutes=1)` cogs (Storm sign-up in `storm_signup_scheduler.py`, Shiny post in `bot.py:shiny_tasks_post_task`, Train/Survey reminders) are two different scheduling paradigms coexisting in the same bot: one dynamic-sleep-until-next-trigger loop for the leadership event-draft flow, and several fixed-per-minute polling loops for everything added since. They're bridged reasonably well by the shared heartbeat/adapter contract in `outage_catchup.py`, so this isn't "competing" scheduling so much as "two coexisting patterns" — but is the long-term intent to migrate the event-draft flow onto the same per-minute polling shape as everything else (simplifying `outage_catchup.py`'s special-cased scheduler exclusion), or is `scheduler.py`'s dynamic-sleep design considered worth keeping because event drafts are comparatively rare and don't need per-minute granularity? Worth a deliberate call rather than letting the split grow by accretion as new features are added.
- Is the missing `pending_warnings` recovery path (Critical finding above) an acceptable gap given how rarely a restart would land in that specific ~5-minute window, or does it need fixing before the next feature is layered on top of the announcement flow? Flagging for prioritization rather than assuming either answer.
