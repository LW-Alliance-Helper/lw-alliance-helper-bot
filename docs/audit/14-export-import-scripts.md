## 2. Summary

This section is generally well-factored and shows real production hardening (the `#348` archived-card fix in `sync_project_status.py`, the `AuthError`-as-soft-skip pattern, the demo-seed's idempotent wipe-then-upsert design). `export_import_cog.py` is a clean UI-only wizard layer — it does **not** duplicate `config_export.py`'s serialization/apply logic, contrary to the risk flagged in the task brief; every read/write of guild config routes through `config_export.*`. The real gaps are: a wizard-registry leak risk in `export_import_cog.py`'s import command (no top-level `try/finally`, contradicting the pattern `wizard_registry.py`'s own docstring prescribes), synchronous SQLite calls issued directly from async command handlers without `asyncio.to_thread`, a boot-time blocking-call risk in `seed_demo.py` when invoked from `bot.py`'s `on_ready`, and a large chunk of copy-pasted view code between `_ChannelRemapView`/`_RoleRemapView`.

## 3. Findings

### Critical

None found.

### High

- **[Error visibility / resource leak] `export_import_cog.py:572-790` `config_import_cmd` registers a wizard-registry session but never wraps the body in `try/finally`.** `cancel_event = wizard_registry.register(user.id)` at line 572 is followed by seven separate `wizard_registry.unregister(user.id, cancel_event)` calls scattered across every early-return branch (lines 582, 589, 660, 691, 722, 726, 743, 747) — but there is no call after the final success path's own return, and more importantly nothing catches an unexpected exception from `config_export.parse_and_validate`, `config_export.apply_import`, or any Discord API call in between. `wizard_registry.py`'s own module docstring (lines 9-21) explicitly documents the `try: ... finally: wizard_registry.unregister(...)` pattern as the intended usage; this cog doesn't follow it. If any call between registration and the final `unregister()` at line 790 raises, the user's wizard slot is never freed — `wizard_registry.is_active(user.id)` stays true and any code gating on it (or a future gate) would incorrectly treat that user as mid-wizard until they explicitly run `/cancel` (which self-heals via `cancel_user`) or the bot restarts.
  - **Recommendation**: wrap the entire body of `config_import_cmd` (from just after `cancel_event = wizard_registry.register(user.id)` through the final report) in `try: ... finally: wizard_registry.unregister(user.id, cancel_event)`, matching the documented pattern, and remove the now-redundant manual `unregister` calls at each early return.

- **[Blocking call in async context] `export_import_cog.py` calls several synchronous, SQLite-backed `config_export.*` functions directly from `async def` command bodies without `asyncio.to_thread`.** `collect_available_categories` (lines 429-433, 486-490), `build_export` (514-521), `parse_and_validate` (586), and `apply_import` (758) all do direct `sqlite3` I/O via `config._get_conn()` (confirmed in `config_export.py`, e.g. lines 330, 408, 453, 500, 552-557, 664, 703, 1145-1478) and are invoked with a plain function call, not `await asyncio.to_thread(...)`. Each call blocks the bot's single event loop for every other guild's commands for its duration. SQLite reads/writes are typically fast, but `apply_import` in particular walks every configured category and writes multiple tables per guild, so this is a real (if usually small) stall on every `/config export` and `/config import` run.
  - **Recommendation**: offload these calls with `await asyncio.to_thread(config_export.build_export, ...)` / `await asyncio.to_thread(config_export.apply_import, ...)` etc., matching the discipline already used elsewhere in the bot for Sheets/DB calls inside async handlers (see `buddy.py`/`buddy_ui.py`'s `to_thread` usage, noted as a positive pattern in the buddy-system audit).

- **[Blocking call in async context — boot path] `scripts/seed_demo.py`'s `seed_demo_guild_from_env()` is called synchronously from `bot.py`'s `async def on_ready`.** `bot.py` (around line 204-210) does `if os.getenv("SEED_DEMO_ON_BOOT") == "1": ... seed_demo_guild_from_env()` with a plain call, not `await asyncio.to_thread(...)`. `seed_demo_guild_from_env()` (this file, lines 548-625) chains `config.init_db()`, seven `config.save_*`/`wipe_demo_extras` SQLite calls, and — unless `SEED_DEMO_SKIP_SHEET=1` — a full `seed_sheet()` pass that clears and rewrites six Google Sheets tabs (network calls to the Sheets API). All of this runs on the bot's event loop during `on_ready`, so on any boot with `SEED_DEMO_ON_BOOT=1` the bot is fully unresponsive (no gateway heartbeats/command handling) for however long the Sheets round-trips take — this is exactly the "blocking gspread calls in async context" case the standards call out.
  - **Recommendation**: wrap the `on_ready` call site in `await asyncio.to_thread(seed_demo_guild_from_env)` (or make `seed_demo_guild_from_env` itself `async` and thread only the synchronous inner calls). Low urgency in the sense that this env var is meant to be toggled briefly and then unset, but the impact when it *is* set is a full bot freeze, not a degraded response.

### Medium

- **[Duplication] `export_import_cog.py:138-269` `_ChannelRemapView` and `_RoleRemapView` are ~90% identical.** Both classes build a "keep current / skip / pick new" wizard step with the only real differences being `discord.ui.ChannelSelect` vs `discord.ui.RoleSelect`, the mention text ("channel or thread…" vs "role…"), and the Discord markdown format (`<#id>` vs `<@&id>`). The `keep_btn`/`skip_btn` construction and callback bodies (lines 176-207 and 238-269) are copy-pasted verbatim aside from the decision-string reuse.
  - **Recommendation**: extract a single `_RemapView` parameterized by `select_factory: Callable[[], discord.ui.Select]`, `entity_label: str` ("channel or thread" / "role"), and a `format_mention: Callable[[int], str]`, and have both `_ChannelRemapView`/`_RoleRemapView` become thin subclasses or be replaced entirely by one class taking those three params.

- **[Duplication] `export_import_cog.py:709-728` (channel-group loop) and `730-749` (role-group loop) inside `config_import_cmd` are structurally identical** — build a numbered prompt string, instantiate the remap view, `wait_view_or_cancel`, handle cancel/timeout the same way, and store the decision in a dict keyed by `grp.source_id`. Only the view class and dict/label differ.
  - **Recommendation**: once `_RemapView` is unified (see above), factor this loop into a shared `_walk_remap_groups(channel, groups, view_factory, cancel_event, unregister_cb) -> dict | None` helper called twice (once per group type), returning `None` on cancel/timeout so the caller can bail with one `if decisions is None: return` instead of two near-identical inline blocks.

- **[Long function / decomposition candidate] `export_import_cog.py:557-790` `config_import_cmd` is ~230 lines** and mixes six distinct phases in one body: permission check, file download + validation, preview embed + confirm, sheet-ID prompt, channel/role remap wizard loop, and apply + report. The phase boundaries are already marked with `# 1) ... # 6)` comments, which makes this a clean decomposition candidate.
  - **Recommendation**: split along the existing numbered-comment seams into private helpers (e.g. `_download_and_validate`, `_confirm_preview`, `_prompt_sheet_id`, `_walk_remaps`, `_apply_and_report`), keeping `config_import_cmd` as the orchestrator that also owns the single `try/finally` for the wizard-registry session (ties into the High finding above).

- **[Duplication] `scripts/seed_demo.py:548-625` (`seed_demo_guild_from_env`) and `631-720` (`main`) run the identical eight-call seeding sequence** (`wipe_demo_extras`, `seed_core_config`, `seed_events`, `seed_train`, `seed_birthdays`, `seed_storm`, `seed_survey`, `seed_growth`, then conditional `seed_sheet` with the same try/except-and-report shape) with only the surrounding print formatting and the args-construction differing.
  - **Recommendation**: extract a shared `_run_seed(args, skip_sheet: bool, log_prefix: str = "") -> int` that both the boot-time and CLI entry points call, so a future new seeding step only needs to be added once instead of kept in sync by hand across two call sites.

- **[Fragile hardcoded config] `scripts/sync_project_status.py:52-62` hardcodes `PROJECT_ID`, `STATUS_FIELD_ID`, and every `STATUS_OPTIONS` GraphQL node ID as module constants**, with a comment noting they should be reconfirmed via `gh project field-list 2 --owner LW-Alliance-Helper` if the project's Status field is ever rebuilt. If that field is ever recreated (not just relabeled), every ID silently goes stale and `set_status` will fail with an opaque GraphQL error rather than a clear "config out of date" message, since `gql()`'s error handling (lines 100-109) only special-cases 401/403 auth failures, not "field/option not found" schema errors.
  - **Recommendation**: no code change required necessarily, but consider having `gql()` detect the specific "could not resolve to a node" / field-not-found GraphQL error text and re-raise with a pointer back to this comment block and the `gh project field-list` command, so a future stale-ID break surfaces as an actionable message in the Actions log instead of a raw GraphQL error dump.

### Low

- **[Dead code] `scripts/upload_storm_emojis.py:33` `logger = logging.getLogger(__name__)` is created and configured via `logging.basicConfig(level=logging.INFO)` (line 121) but never actually used** — every status message in the file goes through `print(...)` instead (lines 67, 86, 96, 102, 111, 114, 127-135). The logger object has no callers.
  - **Recommendation**: either remove the unused `logger`/`logging` setup, or replace the `print()` calls with `logger.info(...)` for consistency with modules elsewhere in the bot that use `logging`.

- **[Redundant import] `scripts/seed_demo.py:569` `import argparse, os` re-imports `argparse`, which is already imported at module scope (line 48).** Only `os` is actually new at that point; the local `import argparse` is a harmless but pointless shadow-import.
  - **Recommendation**: change to `import os` only inside `seed_demo_guild_from_env`, relying on the top-level `argparse` import (used only for `argparse.Namespace`, which module-scope `argparse` already covers).

- **[Naming / structure inconsistency] `export_import_cog.py:632-654` `_ConfirmView` is defined as a class nested inside `config_import_cmd`**, unlike every other view in the file (`CategoryPickerView`, `_ChannelRemapView`, `_RoleRemapView`, `_SheetIdView`), which are module-level. This makes `_ConfirmView` harder to find/reuse and breaks the file's otherwise-consistent "views live at module scope, above the cog" layout.
  - **Recommendation**: hoist `_ConfirmView` to module level alongside the other view classes (it has no closure dependency on `config_import_cmd`'s locals beyond what could be passed via `__init__` or left as-is, since it reads no outer variables at all).

- **[Missing type hints] `scripts/sync_project_status.py`'s module functions have no parameter or return type hints anywhere** (`gql(query, variables=None)`, `get_pr_for_commit(sha)`, `get_closing_issues(pr_number)`, `get_project_item_id(issue_node_id)`, `set_status(item_id, option_id)`). The file isn't imported by other modules today (it's a standalone CI script invoked via the GitHub Actions workflow), so this is low-impact, but it's the only file in this section with zero type hints.
  - **Recommendation**: low priority; add `str`/`int`/`dict`/`bool` hints opportunistically if the file is touched again, e.g. `def gql(query: str, variables: dict | None = None) -> dict:`.

- **[Minor inefficiency] `scripts/sync_project_status.py:188-201` `get_closing_issues` issues one GraphQL request per body-referenced issue number** (inside the `for n in body_nums:` loop) instead of batching via GraphQL aliases in a single query. For the common case (a PR closing 1-3 issues) this is negligible; it would only matter if a PR ever referenced many issues in its body.
  - **Recommendation**: no action needed at current scale; note for later if PRs start closing double-digit issue counts.

- **[Docstrings] `export_import_cog.py`'s command callables (`config_overview`, `config_export_cmd`, `config_import_cmd`) have no docstrings**, relying entirely on the `description=` kwarg passed to `@config_group.command(...)`. This matches the file's own convention (the `description=` *is* the user-facing doc) and Discord surfaces that text directly in the slash-command UI, so this is intentional rather than an oversight — flagged only for completeness per the standards' docstring check.
  - **Recommendation**: none needed; the `description=` kwarg fills this role for Discord-facing commands.

## 4. What's already optimal

- **`export_import_cog.py` genuinely does not duplicate `config_export.py`'s logic.** Every piece of serialization, validation, remap-group discovery, and apply-to-SQLite work (`collect_available_categories`, `build_export`, `serialize_to_json_bytes`, `parse_and_validate`, `discover_remap_groups`, `apply_import`, `CATEGORY_LABELS`, `RemapDecisions`) is imported and called from `config_export.py`, never re-implemented. The cog file is exactly what its own docstring (lines 1-17) claims: "wizard + slash-command glue" with zero business logic of its own. This is the cleanest possible version of the `_cog`/`_logic` split the standards doc asks for, and the two-file overlap flagged as a risk in the audit brief does not materialize in the code.
- **`sync_project_status.py`'s `#348` fix generalizes well.** The archived-card handling in `set_status` (lines 229-254) treats "item is archived" as a recognized, non-fatal GraphQL error and returns `False` so the caller can print a clean "skipped" line (line 317) instead of crashing the whole sync. The same soft-fail philosophy is applied consistently at the top level via `AuthError` (lines 77-81, 321-332): a missing/expired `GH_TOKEN` prints a `::warning::` annotation and exits 0 rather than red-X'ing the whole CI workflow over a secret rotation. Scanning the rest of the file for the same class of problem (an item existing in a state the mutation doesn't expect) didn't turn up another unhandled case — `get_pr_for_commit` and `get_closing_issues` already degrade gracefully (empty node lists, missing PR/issue) rather than assuming presence.
- **`seed_demo.py` and `seed_dev_roster.py` are both genuinely idempotent and say so, and mean it.** `wipe_demo_extras` (seed_demo.py:252-270) explicitly deletes the two tables that would otherwise accumulate across re-seeds (`guild_events`, `guild_extra_surveys`) before the UPSERT-style `save_*` calls run, and `seed_dev_roster.py`'s `--clear` flag + `rng = random.Random(42)` (line 129) make re-seeds byte-for-byte reproducible rather than silently drifting. Both scripts also fail closed on missing setup state instead of writing partial/garbage config (`seed_dev_roster.py:184-191` checks `roster_cfg.get("enabled")` before writing; `seed_demo.py:571-582` checks all required env vars up front and aborts with a clear message rather than crashing mid-seed on a `None`).
- **No hardcoded secrets or credentials anywhere in this section.** Every credential-shaped value (`GH_TOKEN`, `DISCORD_TOKEN`/`DISCORD_BOT_TOKEN`, `GOOGLE_CREDENTIALS_JSON`/`GOOGLE_SERVICE_ACCOUNT_FILE`, `CONFIG_DB_PATH`) is read from an environment variable with no fallback literal, and error messages that include external data (`sync_project_status.py`'s GraphQL error bodies, `export_import_cog.py`'s file-read exception text) don't echo the token/credentials themselves — only request/response content.
- **`check_changelog_slim.py` is a well-scoped, defensively-written utility.** It correctly limits itself to *newly added* lines via a git-diff-against-HEAD baseline (lines 40-70) so historical violations don't drown out new ones, handles the Windows-absolute-path-vs-git-relative-path mismatch explicitly (the comment at lines 51-53 documents a real gotcha it works around), and its `_baseline_lines` catches exactly the exception types it can anticipate (`CalledProcessError`, `FileNotFoundError`, `ValueError`) rather than a bare `except Exception`.

## 5. Open questions

- Should `config_import_cmd`'s per-branch `wizard_registry.unregister` calls (the High finding above) be fixed as a standalone bugfix now, given it's a real leak risk, or bundled with the larger decomposition of the function into phase-helpers? Both changes touch the same code; doing the leak fix first as a minimal `try/finally` wrap and the decomposition later avoids conflating a correctness fix with a refactor.
- `seed_demo.py`'s `seed_demo_guild_from_env()` being called unthreaded from `on_ready` (High finding) has presumably been "fine in practice" because `SEED_DEMO_ON_BOOT` is only set briefly and manually. Is the multi-second-to-tens-of-seconds full bot freeze during that boot an accepted tradeoff (simplicity of the current code) given how rarely it fires, or worth the `asyncio.to_thread` wrap regardless? Kevin's call on priority given it's boot-time-only and self-limiting (operator unsets the env var after use).
- Is there an appetite for a small shared "two-branch remap view" abstraction (the `_ChannelRemapView`/`_RoleRemapView` Medium finding) now, or should that wait until a third remap kind (e.g. category channels, or a future "webhook remap") actually needs it, per a YAGNI read of the current 90%-duplication? Two call sites is a defensible amount of duplication to leave alone; the recommendation above assumes Kevin wants it collapsed regardless, but that's a judgment call, not a standards violation.
